# C++ Trader Backtester — Reading Guide

This project is a good starting point if your goal is to:

- build a usable quant engineering project
- understand low-latency C++ structure
- later use agents to improve performance and software quality

It is not a full production trading system. Treat it as a compact learning project with solid architecture ideas.

---

## Recommended Reading Order

### 1. `README.md`

Read first to understand the project scope: event-driven backtester, pluggable strategies, price-time priority order book, custom memory pool, benchmark-focused C++.

Use it for orientation, but don't take every performance or architecture claim at face value.

### 2. `ARCHITECTURE.md`

Read before touching any code. This file gives the system diagram, component breakdown, data flow (tick processing + order matching), time/space complexity tables, and optimization techniques.

Two things to cross-check against the actual code:

- The doc shows `Tick` as `alignas(64)` with a fixed `char symbol[16]` — the real code uses `std::string symbol` with no alignment. Doc and code differ here.
- Performance numbers in the doc (`33M ticks/sec`, `0.95 ns allocation`) are close to but not identical to what the README claims. Both are benchmark figures, not production guarantees.

Read it for the mental model, not as ground truth.

### 4. `include/types.hpp`

Read before the engine. This file defines the vocabulary of the entire system:

| Type | Notes |
|---|---|
| `Price` | `int64_t`, fixed-point × 10000 |
| `Quantity`, `OrderId`, `Timestamp` | primitive aliases |
| `Side`, `OrderType`, `OrderStatus` | enums |
| `Order`, `Trade`, `Tick` | core structs |

The fixed-point price is a deliberate design choice:

```cpp
using Price = int64_t; // price * 10000
```

Avoids floating-point rounding in the hot path. Deterministic arithmetic matters in trading.

### 5. `include/tick_engine.hpp` + `src/tick_engine.cpp`

The system coordinator. It does not contain trading logic — it only runs the simulation.

Responsibilities:
- receive ticks, maintain order books by symbol
- call each strategy on each tick
- accept orders from strategies, route them into the order book
- notify strategies when trades happen, collect stats

Core event flow:

```
Tick → TickEngine::process_tick
     → find/create OrderBook
     → Strategy::on_tick
     → strategies submit orders
     → OrderBook::match_order
     → trades callback into TickEngine
     → Strategy::on_trade
```

### 6. `include/order_book.hpp` + `src/order_book.cpp`

The most important trading-specific module.

Internal storage:

```cpp
std::map<Price, PriceLevel, std::greater<Price>> bids_;  // highest first
std::map<Price, PriceLevel> asks_;                        // lowest first
```

Matching follows price-time priority: better price first, then FIFO within a price level.

- **Buy limit order:** match against best ask until ask > buy limit; add remainder to bids
- **Sell limit order:** match against best bid until bid < sell limit; add remainder to asks

### 7. `strategies/momentum_strategy.hpp`

Contains `MomentumStrategy` and `MarketMakerStrategy`. Useful for learning the strategy interface, but P&L and position handling are simplified — not financially correct.

### 8. Tests and Benchmarks

Read last, after the core modules:

```
src/test_order_book.cpp
src/test_strategies.cpp
src/test_types_performance.cpp
src/benchmark.cpp
```

These files define what the project actually guarantees. For agent-assisted work, tests and benchmarks are the most important files — they become the judge.

## Architecture Breakdown

### Tick Engine

Market data dispatcher + strategy scheduler + order router + stats collector.

```cpp
void process_tick(const Tick& tick);
void submit_order(const Order& order);
void run_backtest(const std::vector<Tick>& ticks);
void add_strategy(std::unique_ptr<Strategy> strategy);
```

### Strategy Interface

```cpp
virtual void on_tick(const Tick& tick, TickEngine* engine) = 0;
virtual void on_trade(const Trade& trade) = 0;
virtual const char* name() const = 0;
```

Strategies react to market data and trades but submit orders through the engine. This makes them fully pluggable.

### Order Book

Matching logic for limit orders (see §4 above). Uses `std::map`, which gives clean O(log n) logic but is not the fastest possible structure.

### Memory Pool

```cpp
MemoryPool<Order> order_pool_;
```

Avoids `malloc` in the hot path. Pool allocation is close to pointer-bumping: no system allocator, no fragmentation search, predictable latency. Best part of the project if you care about HFT-style engineering.

### Fixed-Point Types

```
150.25  →  1502500  (scale = 10000)
```

Benefits: deterministic calculations, no floating-point surprises, exact comparison in matching logic.

## Performance Claims

The README reports:

| Metric | Value |
|---|---|
| Throughput | 34M ticks/sec |
| Order matching | 7.65M orders/sec |
| Memory allocation | 1.69 ns |
| Tick latency | < 1 µs |

**What these numbers mean in practice:**

- **34M ticks/sec** — measures the in-memory event loop only. No exchange feed decoding, no network I/O, no persistence, no risk system.
- **7.65M orders/sec** — depends on the benchmark order distribution. `std::map` is not the fastest matching structure; treat this as a learning baseline.
- **1.69 ns allocation** — pool allocation, not `malloc`. This is where a memory pool looks its strongest. Not representative of general allocation cost.
- **< 1 µs latency** — benchmark latency, not end-to-end trading latency. Real systems add: market data decoding, risk checks, networking, kernel bypass, logging, clock sync, exchange gateways.

## Known Issues and Code/Doc Mismatches

### `Tick` is not cache-aligned

The architecture doc describes `Tick` as `alignas(64)` with fixed-size symbol storage. The actual code is:

```cpp
struct Tick {
    std::string symbol;
    Price price;
    Quantity volume;
    Timestamp timestamp;
    Side side;
};
```

Doc and code differ. Good future cleanup task.

### Strategy P&L is not order-aware

Strategies don't verify whether a fill belongs to their own orders. The code even notes:

```cpp
// In production, check if trade.buy_order_id or trade.sell_order_id is ours
```

Strategy analytics are demonstration-level only.

### Order routing ignores symbol

`TickEngine::submit_order` routes to the first available order book:

```cpp
order_books_.begin()->second->add_order(order);
```

Orders should carry a symbol or instrument ID. **Strong candidate for your first engineering improvement.**

### `best_bid()` looks like a bug

The bid map is declared descending (`std::greater<Price>`), so `begin()` is the highest bid. But:

```cpp
Price best_bid() const { return bids_.empty() ? 0 : bids_.rbegin()->first; }
```

`rbegin()` on a descending map is the *lowest* bid. Matching logic uses `bids_.begin()` — more consistent. The getter is likely wrong.

## Learning Path

### Phase 1 — Understand and Run

```bash
cd /Users/mac/Desktop/Quant/cpp-trader-backtester-main
cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j
./build/backtester
./build/benchmark
make test
```

Goal: build succeeds, tests pass, benchmark output recorded.

### Phase 2 — Make It More Correct

- Add `symbol` to `Order`
- Route orders to the correct order book by symbol
- Fix or test `best_bid()`
- Make strategy fill handling order-aware
- Improve P&L accounting
- Add regression tests for partial fills

### Phase 3 — Make It More Measurable

- Store benchmark results in `experiments/`
- Add fixed benchmark input
- Record compiler flags and machine info
- Compare before/after performance by commit

### Phase 4 — Agent-Assisted Optimization

Only after tests and benchmarks are stable, give an agent narrow tasks:

- optimize `OrderBook::match_order`
- improve memory pool benchmark
- reduce tick processing overhead
- add missing tests
- refactor strategy accounting

> **Agent rule:** no change is accepted unless tests pass and benchmark results are recorded.
