
# 用Agent榨干低延时最后的性能

## 一句话总结
用一个高频交易撮合引擎的实验说明：性能调优可以被设计成"可测试、可回滚、可迭代"的 AI Agent 闭环。人类的工作重心不再是亲自手写优化，而是搭建让 AI 自主试错和进化的工程沙盒。

---

## 核心观点

低延迟系统的性能优化，本质上是一个反复执行"假设 → 修改 → 测试 → 回滚 → 记录"的循环。这个循环高度依赖工程师经验，极其消耗精力。

作者受 Andrej Karpathy 的 auto research 项目启发，做了一个名为 `auto_optimization` 的实验：给大模型一个订单簿撮合引擎，让它自主进行性能迭代。实验对象是 QuantCup / Tower Research 的夺冠实现——这类代码本身已经极快，优化难度极高。

## 方案设计
项目核心只有两个文件：

- **`evaluate.py`**：自动化裁判，依次执行自动编译、单元测试、压力测试，最终输出一个分数。
- **`program.md`**：AI 的游戏规则，定义优化目标、实验记录格式、测试通过后的 commit 规则，以及失败时的 `git restore` 回滚规则。

每轮循环：AI 提出假设 → 修改代码 → 跑测试 → 看分数 → 记录结论。性能提升且测试通过则 commit，否则回滚。这是一个标准的控制论闭环，也是生物进化论在代码世界里的重演。

---

## 实验结果
接入 Claude Code 后，AI 在约 30 分钟内完成了 20 轮底层迭代，将原本已经极快的撮合耗时降低了约 **23%**。

过程中有几个有意思的现象： 
- **精准局部优化**：AI 发现测试数据中买单占比约 75%，随即加入 `likely` 分支预测提示。看起来是"针对考试"，但从降低延迟的目标来看，无可厚非。
- **并非每次有效**：很多尝试方向偏离重点，收益甚微。但性能指标是客观的——只要测试环境稳定、尝试次数足够，有效尝试终会积累成真实提升。

---

## 设计反思

### 1. 可测性是消除幻觉的前提

没有完善的单元测试，AI 可能为了追求性能破坏业务正确性。实验中就出现过因测试覆盖不足导致的逻辑错误。**可测性是整个闭环能否收敛的基础。**

### 2. 环境噪音会把 AI 带偏

如果压测环境不稳定，AI 无法判断性能变化来自代码改进还是系统抖动，会误入歧途。作者通过 CPU 绑核、进程隔离、实时优先级调度来降低噪音。**没有稳定的压测环境，性能指标就没有可比性。**

### 3. 提示词要提供专家经验

光写"优化性能"没有用。需要定义 AI 的专家角色，并提供数据分布、订单配比、低延迟优化原则等上下文。**指导越具象，AI 寻找突破口的速度越快。**

## 未来方向

- 引入火焰图，让 AI 从盲试升级为根据热点精准打击。
- 脚本化固定流程，减少 Token 消耗，提高调优 ROI。
- 多 Agent 并行探索不同优化路径，类似遗传算法中的多种群演化。

---

## 对交易系统建设的启发

这篇文章的价值不只是"AI 能优化代码"，而是给出了一种研发组织方式的新范式：

- 性能优化从**经验驱动**变成**指标驱动**
- 人肉试错变成**自动化闭环试错**
- "写出一次好的优化"变成**设计一个持续进化的优化系统**

对低延迟交易系统来说，测试框架、压测稳定性、回滚机制和实验记录，未来会和代码本身同等重要。

## 可执行清单

- [ ] 建立覆盖撮合逻辑的单元测试，防止性能优化破坏正确性
- [ ] 固定压测环境（CPU 绑核、进程隔离），排除系统噪音
- [ ] 每次优化必须记录假设、改动、结果、结论
- [ ] 将失败自动回滚、成功自动提交做成规则
- [ ] 在提示词中提供领域约束：订单分布、热点路径、低延迟优化原则
- [ ] 后续引入 perf、火焰图、cache miss、branch miss 等指标，让 AI 从盲试走向精准定位

## 我的理解

低延迟系统过去的壁垒在于专家经验和极细节的工程手感。文章揭示的转变是：当 AI 能持续、大量、低成本地试错时，优势会从"谁更会手写优化"转向"谁更会设计可进化的实验环境"。

**技术壁垒会被算力填平，剩下的壁垒是系统设计能力。**
