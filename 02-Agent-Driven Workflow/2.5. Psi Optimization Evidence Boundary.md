# Psi Optimization Evidence Boundary

## 0. Purpose

This memo defines what counts as evidence for the Psi optimization workflow.

It is not the automatic-loop design, not the roadmap, and not the runtime prompt.

Related documents:

- `2.7. Psi Automatic Optimization Loop Design.md`: loop architecture.
- `2.8. Psi Optimization Inventory and Roadmap.md`: current gaps and next improvements.
- `2.9. Psi Headless Trustee Prompt.md`: copy-paste prompt for a trustee agent.

## 1. Evidence Authority

For Psi, the executable truth is:

- the current Psi source tree
- the active benchmark configuration
- the remote Linux build
- the parquet compare gate
- the timing samples recorded by the harness
- the run artifacts copied back into the local evidence store

External articles, upstream projects, and prior examples may inspire workflow shape.
They do not prove a Psi candidate is correct or fast.

If external material and Psi run evidence disagree, Psi run evidence wins.

## 2. Evidence Types

| Evidence type | Can support | Cannot support |
|---|---|---|
| External article / upstream project | workflow ideas, loop shape, research inspiration | Psi correctness or performance acceptance |
| Psi source code | current behavior, touched paths, possible semantic risk | performance improvement without a run |
| Remote Linux build log | build feasibility under the target environment | correctness or speed by itself |
| Parquet compare output | correctness under the active gate | speed by itself |
| Timing samples | performance comparison under matching conditions | correctness |
| Attempts / timing history | repeatability, noise model, prior failures | claims not represented in raw samples |
| Final report | human-readable summary | replacement for raw artifacts |

## 3. Acceptance Boundary

A candidate can be called `accepted` only when all of these are true:

1. the candidate patch is recorded
2. remote build passed
3. parquet compare passed
4. timing samples were recorded under the same harness
5. the verdict logic says the candidate or stack beat the current control distribution
6. the result row is present in the run artifacts

`neutral`, `rejected`, and `NOISY_PENDING` must be recorded with the same discipline.

## 4. Remote Execution Boundary

Remote Linux is the performance authority, but remote success is not enough on its own.

The local evidence store must contain:

- command or script used
- build log
- compare log
- raw timing samples
- summary statistics
- verdict
- report path

If the result only exists in remote terminal scrollback, it is not accepted evidence.

## 5. Retention Boundary

Evidence should be durable, but not duplicated.

Keep:

- the current run's raw artifacts under the run directory
- the current task's short evidence summary in task notes
- links to the latest report, logs, and charts
- archived task directories after the task is closed

Do not keep in long-lived policy notes:

- per-run medians
- one-off experiment narratives
- copied log blobs
- copied charts or PDFs
- temporary baseline values

When a run becomes stale, archive the task or rotate the run artifacts.
Do not move old runtime facts into the method docs just to preserve them.

## 6. What This Document Must Not Contain

This document must not contain:

- current baseline medians
- historical accepted commit lists
- one-off experiment summaries
- candidate implementation details
- the trustee prompt
- roadmap items that are not evidence rules

Those belong in task notes, run artifacts, `2.7`, `2.8`, or `2.9` depending on the content.

## 7. Bottom Line

Use external material to shape the workflow.
Use Psi build, compare, timing, and artifacts to decide what is true.
