# Psi Optimization Evidence Boundary

## Purpose

This memo defines what counts as evidence for the Psi optimization workflow.

The article in `../01-博客与资料/用Agent榨干低延时最后的性能.md` is useful workflow inspiration. The upstream source project it describes is also useful pattern evidence. Neither one is the executable truth for this repo.

For Psi, the executable truth is:

- the current Psi source tree under `../../Code1/psi-trader-liangjunming`
- the local evaluator in `../scripts/psi_evaluate.ps1`
- the compare tool in `../../Code1/psi-trader-liangjunming/tools/compare_parquet_factor.cpp`
- the baseline parquet set and accepted baseline files in `../../Code1/psi-trader-liangjunming`
- the run logs in `../experiments/psi-evaluator-runs/*/run.md`
- the machine row log in `../experiments/psi_results.tsv`

If the article, the upstream pattern, and Psi disagree, Psi source plus Psi run evidence win.

## Evidence Boundary

| Evidence type | What it can support | What it cannot support |
|---|---|---|
| Article / upstream source project | Workflow shape, experiment cadence, and idea generation | Psi acceptance, Psi performance, or Psi correctness |
| Current Psi source code | Current behavior, local code paths, and what changed in a patch | Claims about improvement without a matching run |
| `run.md` + `psi_results.tsv` | A concrete build / compare / timing outcome for Psi | Claims that were not recorded locally |
| Baseline parquet set | Correctness contract for the current Psi baseline | Performance improvement by itself |
| Remote Linux run | Only after the log, result row, and copied artifacts are back locally | A standalone claim with no local record |

## Locally Verified Claims

These claims are supported by Psi artifacts in this repo.

1. The evaluator is real and local. `psi_evaluate.ps1` can build, run, compare parquet files, record evidence, and write TSV rows.
2. The baseline set is real and fixed. The baseline directory contains 8 parquet files for the Psi comparison set.
3. The compare gate is real and local. `compare_parquet_factor.cpp` is the correctness judge, not `PsiReadWrite::compareFile()`.
4. The accepted baseline is recorded. The qualified baseline run in `2.2. Psi Optimization Experiment Log.md` establishes the current median of 103.225 seconds.
5. The failed attempts are real evidence. Attempts 1 through 4 all passed correctness and then failed the old performance gate.
6. The hotspot profile is local. The logged timing breakdown points at `generateTable`, `WriteTable`, and `readParquet` row fill / stock init as the current large costs.

## Article-Derived But Not Psi Evidence

These ideas are useful, but they remain reference material until Psi itself proves them.

- The closed-loop idea of propose -> edit -> run -> compare -> keep or roll back.
- The separation between an evaluator and a program guide.
- The discipline of treating the benchmark harness as fixed while changing code.
- The advice to record every attempt, including failed ones.

These ideas justify the shape of the workflow. They do not prove a Psi candidate is correct or fast.

## Pending Or Hypothesis-Level Claims

These claims should remain labeled as pending until a local Psi run proves them.

- The remote CentOS 7.9 environment can build Psi without extra manual intervention.
- Arrow / Parquet headers and libraries are exposed correctly on the remote host.
- The Linux baseline run is repeatable on the remote host.
- The next optimization target will produce a stable improvement under the active gate.

## Gate Selection Rules

This memo does not redefine the gate policy. It explains which evidence type supports which gate.

### Class A

Use Class A only when the diff itself rules out a plausible regression mechanism.

Typical signs:

- algorithmic simplification with no new state
- removal of redundant work already proven unnecessary
- using an existing value instead of recomputing it

For Psi, the safe default is Class B whenever the change touches new state, data layout, threading, semantics, or a new build / execution environment.

### Class B

Use Class B for anything that needs empirical confirmation.

This includes:

- new caches or new containers
- semantic changes that should preserve output but may alter behavior
- layout changes in the hot path
- remote execution environment changes

### Bundle / Rebaseline

Use the bundle check as a branch-level drift audit.

It is for cumulative safety, not for deciding a single candidate on its own.

## Remote Execution Traceability

The local Trellis workspace, the local mirror, and the remote execution host must stay aligned.

Rules:

1. Update the local task notes first.
2. Sync the relevant project files to the remote host before running.
3. Record the remote command, timing, and verdict in a local `run.md`.
4. Copy the result row back into `psi_results.tsv`.
5. Do not treat a remote success as authoritative until it exists in local evidence.

## Decision Summary

| Question | Answer |
|---|---|
| Is the article useful? | Yes, as workflow inspiration and pattern evidence. |
| Is the upstream source project useful? | Yes, as a reference for structure and loop design. |
| Is either one the final authority for Psi? | No. Psi source and Psi run artifacts are the authority. |
| Can remote Linux evidence stand alone? | No. It must be copied back into local Trellis evidence. |

## Bottom Line

Use the article and the upstream pattern to explain why the workflow exists.
Use the current Psi source and local runs to decide what is true.
Use remote Linux only as an execution target, not as a separate source of truth.
