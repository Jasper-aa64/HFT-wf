# Host Weather Station and Promotion Gate

## 0. Why This Is A New Phase

The optimization harness has already learned how to do the basic loop:

```text
agent proposes patch -> isolated workspace -> build/correctness -> timing -> ledger
```

The next bottleneck is not patch generation. The bottleneck is measurement
quality. A shared devbox can produce positive timing signals and still be too
noisy to justify promotion.

This phase adds a "weather station" in front of promotion timing:

```text
AI writes the patch;
harness decides whether the measurement weather is credible;
ledger records both patch evidence and host evidence;
promotion only happens under trusted measurement conditions.
```

The goal is to stop treating every completed timing run as equally meaningful.

---

## 1. Problem Statement

Current evidence has two recurring shapes.

First, small Psi candidates often pass correctness and show positive median
speedup, but remain blocked by `NOISY_PENDING`.

Second, the composite Psi stack produced a large positive result:

```text
candidate: stack_skip_unused_row_fields
build: pass
compare: pass
paired_sample_count: 12
median_delta_ms: 8421.500
CI: [7556.500, 9577.000]
p: 0.007996
noise_flag: NOISY
verdict: NOISY_PENDING
```

This is strong discovery evidence, but still not clean promotion evidence.

For TWAP, the first long-run proved that the harness can build, run correctness,
and collect push timing. It did not prove that the agent found a stable
optimization. The artifacts also exposed reporting and verdict-quality issues:
`timing_status=pass` means the tool completed, not that the candidate is good.

Both systems need the same upgrade:

```text
separate patch quality from measurement quality.
```

---

## 2. Research Basis

Performance Roulette describes the same failure mode in system optimization:
noisy measurements can mislead an optimizer, slow convergence, and select
configurations that do not transfer across machines or environments.

The practical lesson is not "run everything more times." More samples can be
expensive and still fail if the sampling environment is poor.

The useful direction is:

```text
modify sampling to improve data quality;
record side-channel environment metrics;
compare candidate and control under the same weather;
promote only when the environment is trustworthy.
```

For this workflow, that means the harness must collect host-condition evidence
beside performance evidence.

---

## 3. Phase Objective

Build a reusable host-readiness layer for optimization runs.

It must answer three questions before a promotion run:

1. Is the host quiet enough to judge performance?
2. If not, should the candidate be parked instead of rejected?
3. Did the ledger record enough environment context for later review?

The expected output is a machine-readable measurement-quality verdict:

```text
QUIET
BORDERLINE
NOISY
```

Only `QUIET` should be allowed to trigger promotion timing automatically.

---

## 4. Weather Station Artifacts

Each audit run should write these artifacts:

```text
host_readiness.json
host_jitter_samples.tsv
host_jitter_summary.json
```

### 4.1 host_readiness.json

Purpose: one current snapshot before running expensive timing.

Suggested fields:

```json
{
  "schema": "host_readiness_v1",
  "recorded_at": "2026-05-26T00:00:00Z",
  "host_key": "17062",
  "decision": "QUIET",
  "active_runner_count": 0,
  "loadavg_1m": 0.0,
  "loadavg_5m": 0.0,
  "cpu_idle_pct": 0.0,
  "cpu_iowait_pct": 0.0,

  "cpu_governor": "performance",
  "cpu_freq_mhz_min": 0.0,
  "cpu_freq_mhz_max": 0.0,
  "disk_read_kb_per_sec": 0.0,
  "disk_write_kb_per_sec": 0.0,
  "blocking_processes": [],
  "notes": ""
}
```

### 4.2 host_jitter_samples.tsv

Purpose: repeated control-only samples used to measure the host, not the patch.

Suggested fields:

```text

recorded_at
kind
duration_ms
loadavg_1m
cpu_idle_pct
cpu_iowait_pct
context_switches_per_sec
active_runner_count

```

The `kind` field should support at least:

```text
snapshot
control_only
baseline_vs_baseline
```

### 4.3 host_jitter_summary.json

Purpose: the actual gate input.

Suggested fields:

## 5. Gate Policy

### 5.1 Promotion Gate

Promotion timing may start only when:

```text
host_readiness.decision == QUIET
and host_jitter_summary.decision == QUIET
and no blocking runner is active
```

If the gate fails:

```text
do not start m24 or promotion timing;
do not reject the patch for performance;
record measurement_quality=low or blocked_by_host_weather;
keep the candidate in prepared / promotion-candidate state.
```

### 5.2 Discovery Gate

Discovery runs can be more permissive, but the ledger must mark the evidence
quality.

Suggested mapping:

| Host Quality | Discovery Timing | Promotion Timing | Ledger Meaning |
|---|---|---|---|
| `QUIET` | allowed | allowed | high-quality evidence |
| `BORDERLINE` | allowed if budgeted | blocked by default | directional evidence only |
| `NOISY` | blocked unless explicit override | blocked | environment not suitable |

### 5.3 Accepted Is Still Strict

This phase does not loosen acceptance.

Automatic promotion still requires:

- build pass
- correctness / compare pass
- clean timing verdict
- measurement quality `QUIET`
- explicit promotion policy allowing the candidate class

Strong positive evidence under `NOISY` should become:

```text
PROMOTION_CANDIDATE_UNDER_NOISY_HOST
```

not `accepted`.

---

## 6. Ledger Additions

Candidate ledgers should record environment context with every timing result.

Suggested fields:

```json
{
  "measurement_quality": "NOISY",
  "host_readiness_path": ".../host_readiness.json",
  "host_jitter_summary_path": ".../host_jitter_summary.json",
  "host_jitter_decision": "NOISY",
  "host_jitter_reasons": [
    "control_cov_above_threshold",
    "active_runner_seen"
  ],
  "promotion_gate": "blocked_by_host_weather"
}
```

This allows later review to distinguish:

- patch correctness failure
- real performance regression
- positive signal measured under poor host conditions
- infrastructure failure
- missing promotion evidence

## 7. Psi Experiment Plan

Do not start by finding another `handlerData.row_loop` micro-candidate.

Use the known strong stack:

```text
stack_skip_unused_row_fields
```

Experiment:

1. Run host weather audit.
2. If the host is `QUIET`, run m24 promotion review for the same unified stack.
3. If the host is `BORDERLINE` or `NOISY`, do not run m24.
4. Record the blocked decision in the ledger and summary.

Expected interpretation:

| Result | Meaning |
|---|---|
| `QUIET` + clean accepted | candidate may enter human promotion review |
| `QUIET` + negative | stack likely not worth promotion |
| `NOISY` + no run | no patch conclusion; host rejected as judge |
| `NOISY` + positive discovery evidence | keep `PROMOTION_CANDIDATE_UNDER_NOISY_HOST` |

---

## 8. TWAP Experiment Plan

Do not continue blind TWAP long-run from static hotspots.

First add runtime stage profiling for the push path:

```text
cache scan
aggregation build
JSON serialization
session fanout
queue push
Redis update path
```

Then run the optimization loop from runtime hotspots, not static guesses.

TWAP timing artifacts must keep these concepts separate:

```text
timing_status=pass      -> test tool completed and no push loss
candidate verdict       -> accepted / neutral / rejected
measurement_quality     -> QUIET / BORDERLINE / NOISY
```

A high-pressure p95 regression is rejected even if `timing_status=pass`.

---

## 9. Implementation Order

Recommended order:

1. Add a standalone host weather audit script.
2. Run calibration audits without any candidate timing.
3. Make Psi promotion timing require weather audit pass.
4. Add ledger fields for measurement quality.
5. Run the gated Psi stack experiment.
6. Add TWAP runtime stage profile.
7. Run a bounded TWAP loop using runtime profile evidence.

This keeps the work incremental. The first milestone is not a faster patch; it
is a trustworthy decision about whether the host is allowed to judge patches.

### 9.1 V0 Calibration Boundary

The first implementation must be a calibration tool, not a hard automation
dependency.

V0 should:

- collect `/proc` host metrics or their remote equivalent
- detect blocking runner processes
- consume explicit control-only or baseline-vs-baseline timing samples
- write `host_readiness.json`, `host_jitter_samples.tsv`, and
  `host_jitter_summary.json`
- classify snapshot-only evidence as `BORDERLINE` at best

V0 must not:

- declare `QUIET` from load average alone
- start m24 timing by itself
- promote or reject a patch
- update Code1, Code2, or any baseline

This boundary prevents the weather station from becoming a fake green light.
Only measured A/A or control-only timing stability can make the host `QUIET`
for promotion.

---

## 10. Stop Conditions

Stop and report instead of running timing when:

- active blocking runner is present
- control-only jitter is above threshold
- baseline-vs-baseline timing is unstable
- host metrics are missing or unreadable
- the candidate has no current correctness evidence
- the candidate is only a discovery probe, not a promotion candidate

The report must say:

```text
no timing started because measurement_quality=<decision>
candidate state unchanged
next valid action=<quiet-window audit / dedicated-host run / runtime profile>
```

---

## 11. One-Sentence Rule

```text
Do not ask a noisy host to make promotion decisions; first measure the weather,
then decide whether the candidate is allowed onto the track.
```
