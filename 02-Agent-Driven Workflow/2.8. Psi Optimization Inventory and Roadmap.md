# Psi Optimization Inventory and Roadmap

## 0. 这份笔记记录什么

这是一份 2.7 之后的简明清单，目的是把当前 Psi 优化工作拆成三层：

- 已经改善 / 已完成
- 仍然 under-modeled / toy-MVP
- 仍然要优化的下一步

它不是新的 gate policy，也不是新的性能结论。

## 1. 已经改善 / 已完成

- remote Linux 已经是性能权威，Windows 不再作为性能证据。
- headless loop 已经是 stateful 的，不再依赖聊天记忆：
  `run_state.json`、`attempts.tsv`、`cooldown.tsv`、`hotspots.tsv`、
  `profile.tsv`、reports、charts 都是流程的一部分。
- 候选队列已经分成三条 lane：`evidence`、`insight`、`combination`。
- gate 顺序已经固定：patch -> remote build -> parquet compare ->
  timing -> attempt verdict -> bundle audit（必要时）。
- 报告产物已经固定为 markdown + PDF。
- `warm/cold` 已经进入证据口径，millisecond timing 也已经保留，seconds 汇总只作为兼容输出。
- `1 warmup + 3 measure` 已经被明确成诊断性流程的一部分，而不是单次感受。
- `NOISY`、`budget_stop`、`no_targets` 这类停止原因已经可以写入状态和报告。
- 候选是否接受，不再只看“感觉像快了”，而是必须经过 build / compare / timing 的完整链路。

## 2. 仍然 under-modeled / toy-MVP

- `same-host / same-dev-machine` baseline jitter 还没有被当作第一类建模对象。
- `3-run` averaging 只能做 screening / triage，不能当作 promotion proof。
- 现在的中心趋势口径还不够稳健，单看 median 不够；
  还需要结合 `MAD` / `IQR`、sample count，必要时再看 trimmed mean
  或 median-of-means。
- baseline 仍然太像“一个全局数字”，而不是条件化后的历史模型。
- 还缺少对 `host_key`、`compatibility_group`、`time_window`、`warm/cold` 的显式联动。
- `NOISY` 现在是有用的状态，但它还太粗，无法区分是同机抖动、时段噪声、样本不足，还是环境漂移。
- 报告里还应该更明确地区分“当前控制组”“同机对照”“历史窗口对照”，否则读者容易把一次诊断结果误读成稳定结论。

## 3. 仍然要优化的下一步

### P0. 把同机抖动建模成第一类对象

- 为每次运行记录 `host_key`。
- 把 `compatibility_group` 作为比较前提，而不是事后解释。
- 把 `time_window`、`warm/cold`、`sample_count` 一起写进证据，而不是只写一个总耗时。
- 比较时优先使用同一台机器、同一配置、同一 gate 形态下的匹配样本。

### P1. 用更稳健的统计口径替代“3 次平均就够了”的直觉

- `median` 继续做主中心趋势。
- 在样本数和分布允许时，再补 `trimmed mean` 或 `median-of-means`。
- 同时记录 `MAD` / `IQR` 作为波动信息。
- `3-run` 平均只保留为 screening 口径，不作为 promotion proof。
- 统计判断只做建模方向，不要写成过度承诺的保证。

### P2. 让历史信息参与候选选择

- `evidence lane` 不只是看当前热点，还要看同机历史、同 gate 历史和最近窗口的噪声形态。
- `insight lane` 的 cache/locality 候选要保留影响阶段、语义风险、回滚条件。
- `combination lane` 要继续做低风险 neutral stack 验证，但不要把“没单点收益”误读成“没机会”。
- 冷却逻辑要带 revisit 条件，而不是永久弃用。

### P3. 让报告更像可复用的工作记录

- 报告正文里要保留“当前控制组 / 同机对照 / 历史窗口”的区分。
- 只在明确标注的示例小节里放基准数值，避免把当天数字写成长期常量。
- 让人能从报告直接看出：这次是在筛查、在建模，还是已经进入可推广证据。

## 4. 口径提醒

这些都属于推荐的建模方向，不是自动成立的结论。

真正的接受条件仍然是：

- remote build pass
- parquet compare pass
- timing evidence recorded
- gate verdict written
- 必要时 bundle audit pass

换句话说，2.8 的目标不是把结论说得更满，而是把“为什么这次算数”说得更稳。
