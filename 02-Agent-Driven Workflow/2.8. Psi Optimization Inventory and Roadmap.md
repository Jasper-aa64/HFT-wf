# Psi Optimization Inventory and Roadmap

## 2026-05-11 Review Evidence Corrections

This section records three concrete gaps between what prior accepted reports
claimed and what the current harness code can actually prove. It exists so
later readers do not re-derive conclusions from evidence that was never
produced by the headless path.

### A. Prior accepted verdicts came from synthetic data, not real candidate evidence

Earlier reports that marked a candidate as `accepted` and attributed a
`0.824s` gain cannot be reproduced from current harness code.

- `psi_auto_optimize.py --dry-run` uses hard-coded `control_samples` such as
  `118.4, 118.9, 118.6` to emit an `Attempt` row.
- The real execution path is `psi_headless_remote.sh` plus
  `psi_headless_longrun.py`. That path neither builds nor times a candidate
  binary. Only the control is measured.
- As a result, any prior `accepted` verdict or numeric gain came from the
  synthetic dry-run, not from a candidate measured on the remote box.

### B. `noise_flag` thresholds are miscalibrated for a 48s workload

`psi_timing_analysis.py` lines 14-16 use absolute thresholds:

- `5ms` range
- `1.5ms` stdev
- `3%` CV

The Psi workload is around `48000ms`. Normal machine jitter on a quiet box
will almost always exceed `5ms` range and `1.5ms` stdev. Once real candidate
timing is wired into the loop, results will pin to `NOISY_PENDING` regardless
of whether the candidate actually helps.

The correct approach is relative thresholds, for example `range / median` or
`stdev / median`. Concrete numbers need calibration against measured jitter;
they are intentionally not fixed here.

### C. Stop contract has unreachable branches

The contract lists `no_targets` and `convergence_proven` as legal stop
reasons, but the current code cannot reach them from the headless path:

- `psi_headless_remote.sh` `next_round_action` only writes `continue` or
  `stop`; it never writes `stop_no_targets`.
- `convergence_proven` only emerges from the synthetic path in
  `psi_auto_optimize.py`, not from `psi_headless_longrun.py`.

Reachable terminal reasons from the real headless path are:

- `budget_stop`
- `remote_failed`
- `repeated_infra_failure`
- `user_stopped`
- `accepted` (which Fix 1 turns off by default, so the loop continues past
  an accepted batch until one of the other reasons fires)

## 2026-05-11 Phase 2 - Timing Rigor Wiring

Second-phase wiring for `05-10-psi-headless-optimization-v2`. Full contract
lives in `.trellis/tasks/05-10-psi-headless-optimization-v2/timing-rigor-spec.md`.

- Interleaved A/B sampling added via `CANDIDATE_RUNNER` env var in
  `psi_headless_remote.sh`; empty means control-only (behaviour unchanged).
- `timing_samples.tsv` gains a `pair_index` column; paired rows use modes
  `paired_control` / `paired_candidate`.
- `psi_timing_analysis.py` noise thresholds are now relative: range/median,
  stdev/median, plus CV. Initial defaults are `2%` / `0.5%` / `2%`. The
  absolute-ms thresholds are retained as a deprecated fallback when
  `control_median_ms <= 0`.
- `summarize_paired_timing` verdict rules: `accepted` requires build pass,
  compare pass, `median_delta > 0`, bootstrap CI low > 0, permutation
  p <= 0.05, noise ok, and `paired_count >= 5`. `NOISY_PENDING` fires when
  the noise flag is NOISY. `rejected` on build or compare fail or
  non-positive median. Otherwise `neutral`.
- `comparison_summary.json` grows `paired_samples`, a `paired` dict, and
  top-level verdict fields (`timing_verdict`, `timing_verdict_reason`,
  `median_delta_ms`, `bootstrap_ci_low_ms`, `bootstrap_ci_high_ms`,
  `permutation_p_value`). The daily report renders them when present.
- `timing_history.tsv` intentionally stays aggregate; raw per-sample pairs
  live in `timing_samples.tsv` + `comparison_summary.json` to avoid a
  schema migration.

## 0. Purpose

This note is the inventory after `2.7`.

It answers three questions only:

1. What is already done?
2. What is still under-modeled or too coarse?
3. What should be improved next?

It is not a new gate policy and not a new loop design.

## 1. Already Improved

- Remote Linux is the performance authority.
- The headless loop is stateful and no longer depends on chat memory.
- The core state surface exists:
  - `run_state.json`
  - `heartbeat.json`
  - `profile.tsv`
  - `hotspots.tsv`
  - `attempts.tsv`
  - `cooldown.tsv`
  - `neutral_pool.tsv`
  - `timing_history.tsv`
- The candidate lanes are split into:
  - evidence
  - insight
  - combination
- The gate order is fixed:
  - patch -> build -> compare -> timing -> verdict -> record
- Reports already separate:
  - current control
  - same-host comparison
  - historical-window comparison
- `1 warmup + 3 measured` is now treated as screening, not promotion proof.
- `budget_stop` and `no_targets` are explicit run states.
- `NOISY_PENDING` remains candidate-level and is tracked through retry conditions, not as a global stop reason.
- `neutral` candidates are preserved for possible stack validation.

## 2. Still Under-Modeled

### 2.1 Same-host jitter

The biggest missing piece is a first-class model for jitter on the same developer machine / same remote host.

That means the evidence model still needs better handling for:

- host-specific drift
- time-window drift
- warm/cold differences
- sample-count dependence

### 2.2 Screening versus promotion

`1 warmup + 3 measured` is useful for fast triage, but too coarse for promotion proof.

Promotion decisions should use more evidence than screening decisions.

### 2.3 Statistical summaries

The current summaries are better than a single wall-clock number, but still not fully rigorous.

The next layer should make these explicit:

```math
\delta = T_{control} - T_{candidate}
```

and for noise:

```math
noise = \max\left(\frac{MAD}{median}, \frac{IQR}{median}, \frac{range}{median}\right)
```

When the sample count is large enough, the roadmap should also support:

- trimmed mean
- median-of-means
- bootstrap intervals
- Welch-style comparison when appropriate

### 2.4 Baseline as a model, not a number

The baseline should be treated as a condition-specific history, not a single global scalar.

At minimum it should be associated with:

- `host_key`
- `compatibility_group`
- `time_window`
- `warm_or_cold`
- `sample_count`

### 2.5 Noise classification

`NOISY` is useful, but it is still too coarse.

It should be split carefully into:

- machine jitter
- window noise
- insufficient sample count
- environment drift

### 2.6 Evidence-lane ranking formula drift

`2.7` and `2.6` still describe the ranking formula as `s = c * o * r * l`.
That expression is no longer the default in code.

- `psi_control_loop.py` default sort key is `score_evidence`, computed from
  `expected_delta_seconds`, `p_owned`, `p_safe`, `p_gate`, `p_local`,
  `cost_attempt_seconds`, `uncertainty`, and `lambda`.
- The old product is retained only as `legacy_corl_score` for compatibility.

The doc and code should be reconciled. The preferred direction is to update
`2.7` to describe `score_evidence` as the current ranking expression, not to
roll code back to the legacy product. Ranking-field changes go in `2.7`; this
note only records the drift.

## 3. Next Priorities

### P0

Make same-host jitter a first-class modeling target.

### P1

Upgrade the statistical summaries so promotion evidence uses more than `median` alone.

### P2

Make historical context part of candidate selection:

- previous attempts
- cooldown state
- prior neutral stacks
- recent noise pattern

### P3

Make the report more reusable:

- always show current control
- always show same-host comparison
- always show historical-window comparison
- always show the reason for the verdict
