# Psi Optimization Harness Operator Manual

Date: 2026-05-28

This is the practical operator manual for the Psi optimization harness. It is
written for a new agent or human operator who needs to continue the optimization
work without reading the whole chat history.

## 1. What This Harness Is

This is an agent-supervised performance optimization workflow. The AI agent is
the headless operator: it proposes patches, builds them, runs the verification
loop, interprets artifacts, records verdicts, and keeps searching for remaining
performance.

The human is not the patch-by-patch executor. The human intervenes where the
task is genuinely under-specified or business-dependent:

```text
what is worth optimizing
which interface or workflow matters
which scale exposes the real bottleneck
which semantics must not change
whether to widen scope or stop squeezing
whether to promote a result into business code
```

Once those boundaries are clear, the agent should run the engineering loop
without asking for approval on every candidate.

The harness is a closed loop:

```text
candidate idea
  -> isolated workspace
  -> patch command / patch agent
  -> remote build
  -> correctness compare
  -> paired timing
  -> verdict
  -> ledger / retry / neutral / accepted state
```

Its job is not just to find faster code. Its job is to avoid accepting a patch
that only looks fast because the measurement window was noisy, incomplete, or
semantically unsafe.

The goal is to use the agent to squeeze out as much safe performance as
practical, while making the evidence strong enough that the human can trust the
delegation.

The user should not manually review every candidate. The expected operating
model is:

```text
human sets strategic boundaries
agent generates and applies candidate patches
harness runs bounded build / correctness / timing / E2E checks
agent records machine-readable artifacts and verdicts
human reviews batch summary or promotion decision
```

If the workflow requires the user to approve every patch one by one, the
workflow has regressed from headless mode.

## 4. The Two Operating Modes

### 4.1 Discovery Long-Run

Purpose:

```text
Find possible optimization candidates.
```

Discovery is allowed to reject many candidates. It should not require a clean
bare-metal host for every attempt. A noisy discovery result should be recorded,
not treated as final promotion evidence.

The agent is expected to choose and revise tactics inside the approved target
area. For a business-interface task, that can include selecting a scale that
amplifies the suspected bottleneck, splitting the work by endpoint, writing a
temporary E2E tool, and iterating after each measured bottleneck.

Discovery can run with bounded budgets such as:

```text
max-candidates = 5 or 8
measure-runs = small or moderate
```

Expected user involvement:

```text
confirm target / scale / forbidden semantics
start batch
wait
read summary
```

The user should not approve every candidate.

### 4.2 Promotion / Replication Review

Purpose:

```text
Decide whether a strong candidate is safe enough to promote.
```

Promotion requires stricter evidence:

```text
correctness compare passes
paired A/B evidence exists
validation lock is active
remote harness tree is known
artifacts are copied back
verdict is accepted / accepted_noisy_replicated / accepted_class_a
```

Promotion must not use:

```text
partial runs
contaminated runs
runs without comparison_summary.json
runs where attempts.tsv only has a header
runs where the validation lock was absent
```

## 5. Verdict Vocabulary

### accepted

Clean or quiet evidence. Candidate can be considered for promotion.

Meaning:

```text
statistically strong
compare pass
noise acceptable
can update promotion state after manual review
```

### accepted_noisy_single

Strong positive evidence from one noisy run.

Meaning:

```text
do not call it failed
do not update baseline
do not push business code automatically
revert patch after run
queue replication / later validation
```

This state exists because a strong noisy signal is still useful evidence.

### accepted_noisy_replicated

Multiple independent locked noisy runs point the same way.

Meaning:

```text
shared-host promotion evidence
artifact must say non-bare-metal / shared-host
still requires explicit human promotion decision
```

### accepted_class_a

Correctness-only acceptance for explicit Class A changes.

Allowed only when the hard whitelist validates the change as a pure low-risk
removal or simplification, such as dead-store deletion.

### NOISY_PENDING

Measurement is too noisy and the direction is not accepted yet.

Meaning:

```text
candidate-scoped state
do not stop the whole run
do not update baseline
record retry condition
```

### neutral

No clear benefit or too weak to promote, but not harmful enough to reject.

Meaning:

```text
revert patch
optionally keep as neutral stack material
do not update baseline
```

### rejected

Bad candidate, correctness failure, clear regression, unsafe semantics, or
unusable evidence.

Meaning:

```text
revert patch
ledger should block obvious repeat
```

### infra_blocked

The harness or environment failed, so the candidate was not judged.

Examples:

```text
control runner missing
remote sync failed
host already busy
control itself fails
```

## 8. Sample Count Contract

`MEASURE_RUNS` means paired A/B evidence budget:

```text
paired_control count
paired_candidate count
```

`NO_COMPARE_RUNS` means candidate-only health-smoke budget before compare.

For promotion replication, use:

```text
MEASURE_RUNS=24
NO_COMPARE_RUNS=1
```

This means:

```text
1 warmup + 1 measured no_compare candidate smoke
1 compare
24 paired control runs
24 paired candidate runs
```

Do not use:

```text
MEASURE_RUNS=24
NO_COMPARE_RUNS unset
```

because legacy behavior makes `NO_COMPARE_RUNS=MEASURE_RUNS`, causing 24
candidate-only no_compare runs that do not contribute to the paired verdict.

Relevant HFT-wf commit:

```text
0ffc00b fix: split psi no-compare smoke runs
```

## 9. When To Run Long-Run

Run long-run when:

```text
the goal is discovery
the user does not need immediate promotion
candidate ledger blocks known bad repeats
the run is bounded by max-candidates or max-hours
the devbox is available
```

Do not stop discovery just because one candidate is noisy. Record the verdict
and continue.

Long-run should not ask the user about every candidate. It should produce a
batch summary.

## 10. When Not To Run Long-Run

Do not start long-run when:

```text
the devbox is needed by another task
remote /root/work/HFT-wf is stale and no clean remote_hft_root is provided
there are active PsiTraderRunner or psi_headless_remote processes
the validation lock is held by another task
the goal is promotion of one already-known candidate
```

If the goal is promotion, run a targeted prepared-candidate review instead of a
free candidate search.

## 11. User Involvement Boundary

The user should participate in high-uncertainty framing:

```text
which business objective matters
which representative scale should expose the issue
which interfaces must be covered
which side effects are forbidden
whether to continue squeezing after the first bottleneck is removed
whether a promotion-grade result should land in business code
```

The agent should own the engineering execution:

```text
inspect code and profile evidence
propose patches
materialize candidate workspaces
build and run remote verification
create or adapt scenario-specific test tools when needed
inspect artifacts and find the next bottleneck
reject / neutral / accepted_noisy candidates according to the gate
clean up test data and stop runners
write the final summary
```

The human should not become the patch queue manager. If every candidate needs a
human yes/no, the headless workflow has failed.

```text
optimization process log path
convergence chart path or the raw data needed to draw it
```

It must not say "optimized" just because a batch ran.

## 16. Optimization Process Record And Convergence Report

The headless agent must leave enough process data to reconstruct the optimization
story. A final chat summary is not enough.

For each meaningful iteration, record at least:

```text
iteration
candidate_id
changed files
hypothesis
patch summary
build result
correctness / E2E assertions
main timing metric before
main timing metric after
delta
verdict
next bottleneck or stop reason
artifact paths
```

For business-interface optimization, the process record should also include:

```text
scenario scale
endpoint / action
DB state before and after
async wait result, if any
test data cleanup status
```

The final report should include a convergence view. For Psi paired timing this
can be the best accepted / accepted-noisy delta by iteration. For service E2E
work such as TWAP aggregation, it can be endpoint wall time by patch round, for
example:

```text
baseline stopAggregation 187.5s
patch sql batch select 187.5s -> still bottlenecked elsewhere
patch batch update 5.2s
patch withdraw batch insert validates sibling endpoint
```

The chart can be generated as an image, a markdown table, or both, but the raw
TSV/JSON data must be kept under the run/task artifact directory so another
agent can regenerate it.

Minimum expected artifacts:

```text
optimization_process.tsv or optimization_process.jsonl
convergence_data.tsv or convergence_data.json
final_report.md
charts/<run_id>_convergence.*
```

If chart rendering is unavailable, write the convergence data and say chart
rendering was not run. Do not omit the process record.

## 17. Common Failure Modes

### Long-run seems stuck

Check whether `MEASURE_RUNS` is being reused as `NO_COMPARE_RUNS`. If yes, the
run may be doing many candidate-only no_compare executions before paired A/B.

### Remote result looks valid but local attempts.tsv is empty

The batch did not return or propagate final artifacts. Do not count it.

### timing_status=pass

This means the timing tool ran. It is not an optimization verdict.

### Strong positive but NOISY

Do not call it failed. Use `accepted_noisy_single` if the positive signal is
statistically strong but noisy.

### Active runner during promotion

Promotion evidence is invalid unless the active runner is explicitly part of
the controlled run and the validation lock policy allows it.

## 18. Mental Model

Discovery is a search engine.

Promotion is a court case.

The harness should search broadly, reject automatically, preserve strong noisy
signals, and only demand strict clean evidence when a patch is about to affect
the baseline or business code.
