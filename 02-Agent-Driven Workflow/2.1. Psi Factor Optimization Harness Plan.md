# Psi Factor Optimization Harness Plan

This is the concrete second phase for `0. Agent-Driven Workflow.md`: use an agent-controlled optimization harness to improve the current Psi factor pipeline, then write Blog 2 from the real run.

The direction is the same as `auto_optimization`:

```text
program.md  -> rules for the optimization agent
evaluate.ps1 -> deterministic judge: build, run, compare, measure
results.tsv -> attempt log
git commit / git restore -> keep-or-revert protocol
```

The important correction from the earlier discussion:

`PsiReadWrite::compareFile()` is not enough for the harness. It logs mismatches but does not return a failing exit code. The automatic correctness gate should use `tools/compare_parquet_factor.cpp`, because that tool returns `0` on match and non-zero on mismatch.

## 1. Target Project

Target repo:

```text
C:\Users\liangjunming\Desktop\work\Code1\psi-trader-liangjunming
```

Primary hot path for the first deterministic optimization:

```text
PsiFactorPipline/PsiReadWrite.cpp
timestampUsToInt64()
```

Observed code behavior:

- `readParquet()` calls `timestampUsToInt64()` once per input row.
- The original function converts every timestamp through `localtime_s` / `localtime_r`.
- The function also had an unused default `std::string format` parameter, creating avoidable call overhead.
- A local cache by second is valid because the formatted base only changes when the timestamp second changes; milliseconds are appended afterward.

First attempted optimization:

```text
cache formatted YYYYMMDDhhmmss base per second
return cached_base + millisecond_part
```

This was a reasonable hypothesis, but the evaluator rejected it in the first real pressure test. It preserved correctness but slowed the measured full-run median from `103.225s` to `106.854s`, so the candidate was reverted. See:

```text
02-Agent-Driven Workflow/2.2. Psi Optimization Experiment Log.md
```

## 3. Correctness Gate

Use file-level parquet comparison:

```powershell
compare_parquet_factor.exe <candidate.parquet> <baseline.parquet>
```

Expected behavior:

```text
exit 0 -> exact enough match
exit 1 -> mismatch
exit 2 -> usage error
```

The current compare tool uses:

```text
key = date|time|thscode|factor_type
tolerance = 1e-8
```

This is stronger than `PsiReadWrite::compareFile()` for automation because the exit code is machine-readable.

## 4. Performance Gate

Use PowerShell because the target environment is Windows and the runner is a Windows `.exe`.

First evaluator policy:

```text
1 warmup run, not counted
3 measured runs
median wall-clock seconds
accept if:
  correctness passes
  median_after <= median_baseline - 5 seconds
```

CPU pinning can be added after the basic loop works. Do not block the first harness on pinning.

## 5. Agent Loop

The optimization agent receives `program.md` and must follow this loop:

```text
1. Write one hypothesis.
2. Modify only allowed production files.
3. Run scripts/psi_evaluate.ps1.
4. If pass:
   - append result row
   - commit with `perf: ...`
   - update accepted baseline timing
5. If fail:
   - git restore allowed files
   - append failure row
   - try a new hypothesis
```

For the first real loop after the timestamp cache, allowed target should be narrow:

```text
PsiFactorPipline/PsiReadWrite.cpp
PsiFactorPipline/PsiReadWrite.h
```

Next targets (profile-guided, from `2.2. Psi Optimization Experiment Log.md`):

Profile breakdown (single diagnostic run, 2026-05-07):

```text
handlerData row_fill:   ~29.5 s total  (stock_init ~15.4 s inside)
generateTable append:   ~30.1 s total  (4 attempts exhausted this path)
WriteTable:             ~9.9 s
```

`generateTable` has been attacked 3 times (Attempts 2, 3, 4) with no accepted result. Do not continue there. Next productive target is `readParquet` → `row_fill` / `stock_init`, where ~15.4s of per-stock initialization repeats for every new parquet day. See `2.2` for details.

## 6.1 Workspace Topology

Use a split layout:

- Local `C:\Users\liangjunming\Desktop\work\` is the control plane.
- Keep Trellis, workflow docs, sync scripts, and a lightweight mirror of the active project locally.
- Remote `/root/work/` is the execution plane.
- Run builds, evaluations, and large data jobs on the remote machine.
- Sync code and docs to the remote machine before execution.
- Pull logs, baselines, and experiment evidence back to the local workspace after each run.

Do not delete the local project mirror for an active optimization workflow. If the local tree only contains Trellis files, the agent can read the rules but cannot inspect the code it needs to optimize.
