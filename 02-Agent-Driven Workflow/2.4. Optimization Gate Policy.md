# Optimization Gate Policy

This document fixes the gate policy after four attempts on the Psi factor harness exposed a convergence problem with the original `improve >= 5s` rule. It replaces section 4 of `2.1. Psi Factor Optimization Harness Plan.md`.

---

## 1. Evidence That Forced This Revision

Four attempts so far, all `FAIL_PERF`:

| Attempt | Target | Improve | Verdict | Observed signal |
|---|---|---|---|---|
| 1 | `timestampUsToInt64` per-second `thread_local` cache | -3.629s | FAIL_PERF | observed slower (-3.629s); inconclusive under 3-run noisy gate |
| 2 | `generateTable` fixed-buffer append | -2.765s | FAIL_PERF | observed slower (-2.765s); inconclusive under 3-run noisy gate |
| 3 | `generateTable` static date/time/thscode columns | +1.851s | FAIL_GATE | observed faster (+1.851s); below old acceptance margin |
| 4 | Attempt 3 + `factor_type` scalar array | -0.359s | FAIL_PERF | near neutral (-0.359s); no reliable signal |

Attempt 3 is the strongest positive observation so far. Correctness passed, and wall-clock moved from `103.225s` to `101.374s` (median of 3), but the harness still reverted it because `1.851 < 5`. That is evidence of improvement, not proof that the gain would hold under broader conditions. Attempt 4 is near neutral and does not give a reliable signal that the earlier win was durable.

Workstation wall-clock variance during this session was approximately `±4-7s` on a `~103s` run. The `5s` threshold is therefore heavily biased toward false rejects.

---

## 2. Why The Fixed-Threshold Rule Is Wrong For This Project

A perf gate balances two errors:

```text
Type I (false reject): real win below threshold, candidate reverted
Type II (false accept): random luck looks like a win, drift accumulates
```

Setting the threshold means picking which error you tolerate. With `±5s` noise:

- threshold `5s` → almost no Type II, lots of Type I (current state)
- threshold `0s` → no Type I, drift via Type II
- threshold `2s` → trades one for the other, no real progress

The convergence question raised in discussion is correct:

```text
If each later optimization is necessarily smaller than the noise floor,
a fixed threshold guarantees the workflow stalls.
A workflow that only works for the first big cut is not really a workflow.
```

The right answer is not "pick a different number for `MinImproveSeconds`". It is "stop using a fixed-threshold rule for changes whose correctness is provable from the diff alone".

---

## 3. What Real Systems Do

Worth grounding the policy in established practice rather than inventing it.

```text
Linux kernel:
  most micro-optimizations land on code review + algorithmic reasoning
  0day kernel test bot runs benchmarks on a separate cadence
  regression detection runs over long windows, not per commit

Chromium:
  perf_waterfall measures every commit
  statistical bisect triggered when a regression is detected
  measurements logged, not gated

Google Benchmark, JMH, Criterion (Rust):
  multiple iterations + statistics
  Welch's t-test or bootstrap confidence intervals
  significance at p < 0.05 instead of a hand-tuned threshold
```

Reference for the statistical approach:

```text
Georges, Buytaert, Eeckhout.
"Statistically Rigorous Java Performance Evaluation." OOPSLA 2007.

Key point: comparing means with a fixed threshold misclassifies under
realistic noise; significance testing is the standard fix.
```

None of these systems use a per-commit fixed-threshold gate as the only mechanism. They all rely on either (a) reasoning at review time for safe refactors, or (b) statistical comparison instead of fixed thresholds, or (c) batch / windowed regression checks. The new policy below uses all three.

## 4. Decision: Two-Class Policy + Bundle Check

### Class A — algorithmic certainty

Definition:

```text
The change is a logical refactor with no plausible regression mechanism.
- no new state visible to callers
- no new cache or branching layout that callers depend on
- no new memory allocation pattern in the hot path

Examples:
- inner-scan O(n^2) replaced by O(n) running variable
- removal of redundant computation already done elsewhere
- using a value already in the right type instead of constructing a copy
```

Counterexample (important):

```text
A thread_local per-second cache is NOT Class A.
It introduces TLS access cost and a per-call branch.
That is exactly why attempt 1 failed.
```

Gate for Class A:

```text
correctness:  required (compare_parquet_factor exit 0 on all files)
perf:         recorded but not gated
```

### Class B — empirical change

Definition: anything not in Class A. New state, new data layout, swapped containers, new threading, anything where reasoning alone cannot rule out a regression.

Gate for Class B:

```text
correctness:  required
perf:         Welch's t-test, p < 0.05, candidate mean < baseline mean
```

The statistical test adapts to current noise. On a noisy day it demands a larger gap before accepting; on a clean day it accepts smaller wins. Operator never tunes a threshold by hand.

### Bundle verification

After every `N` accepted Class A changes, or before merging an experiment branch:

```text
7 measured wall-clock samples
compare to original_baseline.txt (snapshotted at branch start)
branch-level rebaseline audit: flag if the 7-sample median fails the original >= 5s improvement target
```

This is not a primary candidate-acceptance gate. It is a branch-level safety net for cumulative drift after accepted Class A changes, and if it fails the branch should stop for human review before merge. If a Class A change was misclassified and silently slowed the system, the bundle check catches that drift before it leaves the branch. `N = 5` is a reasonable starting value.

---

## 5. Why This Specifically Solves The Convergence Problem

```text
Old policy:   each candidate competes against ±5s noise individually
              small real wins are statistically indistinguishable from 0
              the workflow stalls once individual gains drop below 5s

New policy:   Class A wins land on reasoning (no perf comparison needed)
              Class B wins land on statistical significance (adapts to noise)
              cumulative state is checked against original baseline periodically

Result:       a chain of +2s Class A wins can compound to +10s.
              if the chain silently drifted, bundle check rejects the whole
              experiment branch instead of one commit.
```

The shift is from "every commit must be locally provable" to "every commit must be locally safe, and the branch as a whole must be globally improving".

## 6. Concrete Harness Contract

The executable entrypoint is now:

```text
scripts/psi_evaluate.ps1
```

It is PowerShell so it can run both on Windows and on Linux with PowerShell 7
(`pwsh`). The Linux remote still needs the build environment from the active
remote-readiness task; this policy only defines the gate once the runner and
compare tool exist.

Implemented gate switches:

```powershell
-Class algorithmic   # correctness only; perf recorded but not gated
-Class empirical     # correctness + Welch's t-test (default)
-Rebaseline          # 7 samples vs original_baseline.txt; branch audit / pre-merge check
```

Behavior matrix:

| switch | correctness | perf gate | samples | comparison target |
|---|---|---|---|---|
| switch | correctness | perf gate | samples | comparison target |
|---|---|---|---|---|
| `-Class algorithmic` | required | record-only | 3 | informational vs `baseline.txt` |
| `-Class empirical` (default) | required | Welch p<0.05 and candidate mean < baseline mean | 5 | `baseline.txt` |
| `-Rebaseline` | required | branch audit; fail if median improvement < 5s | 7 | `original_baseline.txt` |

New file:

```text
original_baseline.txt
  snapshotted once at the start of an experiment branch
  never overwritten until branch is merged or abandoned
  consumed only by --rebaseline
```

`baseline.txt` continues to be the rolling accepted baseline (updated after every accepted candidate, A or B).

Welch's t-test is implemented by `scripts/psi_welch_test.py`, called by the evaluator:

```text
inputs:  candidate samples (n=5), baseline samples (n=5)
compute: x_bar, y_bar, s_x^2, s_y^2
         t = (x_bar - y_bar) / sqrt(s_x^2/n_x + s_y^2/n_y)
         df via Welch-Satterthwaite
         p (two-tailed) via Student-t cdf approximation
decide:  accept if p < 0.05 and x_bar < y_bar (candidate is faster)
```

The helper uses a regularized incomplete beta implementation for the Student-t
CDF and has no third-party dependency.

Examples:

```powershell
# Class A candidate.
powershell -ExecutionPolicy Bypass -File scripts\psi_evaluate.ps1 `
  -WithBuild -WithCompare -Class algorithmic

# Class B candidate.
powershell -ExecutionPolicy Bypass -File scripts\psi_evaluate.ps1 `
  -WithBuild -WithCompare -Class empirical

# Bundle audit before merge.
powershell -ExecutionPolicy Bypass -File scripts\psi_evaluate.ps1 `
  -WithBuild -WithCompare -Rebaseline
```

Linux remote example:

```powershell
pwsh -File scripts/psi_evaluate.ps1 `
  -Repo /root/work/Code1/psi-trader-liangjunming `
  -BaselineDir /root/work/Code1/dataset/baseline/psi-factor-20140102-20140103 `
  -OutputDir /root/work/Code1/dataset/output `
  -WithBuild -WithCompare -Class empirical
```

If the build layout differs, pass explicit `-RunnerPath`, `-ComparePath`,
`-RunnerWorkDir`, or `-BuildCommand`.

Each run writes:

```text
candidate result file: <Repo>/<BaselineFile>
human evidence:        experiments/psi-evaluator-runs/<timestamp>/run.md
machine row:           experiments/psi_results.tsv
```

## 7. Application To Current Pending Candidates

Reclassify the three proposals in `docs/4.WorkArrangement/5_性能优化提案.md`:

| Candidate | Class | Rationale |
|---|---|---|
| `timestampUsToInt64` per-second cache | B | introduces TLS state and a per-call branch; failed empirical gate at `-3.629s`; do not retry under Class A |
| `AvgVolCmp::afterCompute` O(n²) → O(n) LOCF | **A** | replaces inner backward-scan with a single `lastValue` accumulator; same LOCF semantics; no new state visible outside the function |
| `getKlineIndex` use existing `m_mapIndex` | **A** for init; **separate change** for the `index > 1` bug fix | hash lookup replaces linear scan, no new state semantics; the bug fix changes which ticks are flagged invalid and must be tested as a behavior change, not a perf change |

Recommended next loop:

```text
1. AvgVolCmp Class A change.
   evaluator: --class algorithmic
   correctness pass -> keep, append result row, perf is informational

2. getKlineIndex m_mapIndex init only (no behavior change).
   evaluator: --class algorithmic
   correctness pass -> keep, perf is informational

3. getKlineIndex bug fix as separate change.
   evaluator: --class empirical (this is a semantics change)
   correctness pass means: 8 parquet outputs unchanged from accepted baseline
   if outputs change, escalate to human review (it may be the existing bug
   was being relied on; needs reasoning, not auto-accept)

4. After three accepted Class A keeps, run --rebaseline.
   7 samples vs original_baseline.txt (median = 103.225s)
   use the 5s target as a drift flag, not a candidate-acceptance gate
   if pass: proceed with normal branch review
   if fail: stop merge and investigate cumulative drift across the Class A
           changes
```

## 8. What This Policy Does And Does Not Solve

Solves:

```text
- real small wins (attempt 3 type) are no longer discarded
- the workflow can converge past the noise floor when the diff is provably safe
- cumulative drift is caught by the bundle check before merge
- threshold tuning by hand is replaced by statistical significance
```

Does not solve:

```text
- noise floor on this workstation
  closing background apps, high-performance power plan, and disabling
  Defender real-time scan on the dataset directory remains the highest-
  leverage independent improvement
- classification mistakes
  if a Class B change is wrongly tagged Class A, only the bundle check
  catches it. Be conservative about Class A. When in doubt, use Class B.
- write-phase optimization
  generateTable + WriteTable can account for a large share of the baseline.
  this policy does not change which targets are worth attacking, only
  how the harness judges them
```
