## Implemented vs target automation

This plan separates the reliable execution loop that already exists from the
fully automatic optimization loop that remains a target state.

Implemented today:

```text
remote build/compare/timing -> profile/control-loop artifacts -> evidence/insight/combination lanes -> human-or-agent candidate choice -> gate result -> performance optimization report
```

Target automation:

```text
profile run -> DIAGNOSTIC_ONLY record -> machine-readable hotspots -> candidate queue -> patch proposal -> build -> correctness -> timing -> attempts/cooldown update -> next iteration
```

The current system is therefore not "unimplemented"; it is a reliable,
agent-assisted optimization harness with a clear path toward more autonomous
patch generation and selection.

## Headless scope

This run should not push remote commits automatically.

It may edit and test local/remote working trees, but accepted code must be reported before pushing unless the user explicitly authorizes auto-push.

Prefer visible Codex worker batches with bounded commands and a clear status
surface. Each batch should state its scope, expected artifacts, and terminal
result format before it starts. Do not launch invisible detached remote work
such as `nohup`, `tmux`, or `screen` jobs unless the user explicitly approves
it and the run publishes a visible status artifact that can be checked while it
is running.

Do not make the main thread wait behind one long synchronous `wait_agent` call
when a sub-agent is running. Use short status checks, mailbox updates, or
user-triggered status queries instead, and answer user questions directly even
while background work is still in flight.

### Step 4: Rank next targets

Use this scoring rule:

```math
s = c \times o \times r \times l
```

Where `c` is observed cost, `o` is ownership confidence, `r` is correctness
safety, and `l` is locality.

Treat this ranking as the evidence lane only. It is not a greedy-only rule and
it is not the full search strategy.

Maintain three candidate lanes:

```text
evidence lane: profile/hotspots ranked by observed cost, safety, and locality
insight lane: agent-proposed Class A or cache/locality candidates, including small non-top-hotspot gains with a strong rationale
combination lane: neutral-stack validation for individually neutral, low-risk candidates
```

Do not repeatedly retry exhausted targets unless there is new evidence.

Do not make the stack greedy-only. Keep one evidence lane for the top safe
hotspot, one insight lane for model-generated Class A/cache-locality candidates,
and at least one combination lane for a low-risk neutral stack. Cooldowns should
suppress known dead ends or broad targets until a narrower hypothesis exists.

Cache/locality candidates can target a non-top timed stage when they reduce
repeated access, improve data locality, reduce pointer chasing or allocation, or
reuse hot data. Record the affected stage and semantic risk before testing.

Class A still requires the correctness gate and bundle drift audit. It means one
noisy performance sample should not reject the candidate by itself; it does not
mean guaranteed no regression or no testing.

The harness should constrain verification quality and evidence, not suppress
reasonable model-generated hypotheses as models get stronger.

Known target status belongs in `attempts.tsv`, `cooldown.tsv`, run notes, or
performance optimization reports. Keep this plan focused on the schema, not
historical winners.

```text
target:
status: accepted / blocked / cooldown / neutral / investigating
evidence artifact:
reason:
revisit condition:
```
