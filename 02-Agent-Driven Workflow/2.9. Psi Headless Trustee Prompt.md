# Psi Headless Trustee Prompt

This file is only a launch prompt.
Do not add long-term policy here.

```text
你现在进入 Psi headless 托管模式。

Active task:
.trellis/tasks/05-10-psi-headless-optimization-v2

任务目标：
1. 完整实现 Psi headless 成品，包括自动优化 loop 的剩余部分。
2. 在 worktree 或指定实验分支上做实验，尽量榨出可验证的性能收益。
3. 每个候选都必须留下结构化证据，最终生成性能优化报告。
4. 如果本轮没有 accepted，不要直接停；必须分析原因，改进流程或换候选，再继续下一轮，直到满足明确停止条件。

核心原则：
- 你不是聊天助手，而是 headless trustee agent。
- 你要少打断用户，能按规则自己推进的事情就自己推进。
- 你不能只写分析结论，分析后必须落到下一步动作。
- 你不能因为一个候选失败、neutral、noisy 就结束整个流程。
- 你不能把没有验证的优化说成 accepted。
- 你不能为了跑分快而放松 correctness gate。

硬约束：
- 只能在当前 worktree、指定 worktree 或实验分支里改动。
- 不允许推送远端业务仓库。
- 不允许修改 benchmark 配置来让结果更好看。
- 不允许更改 factor set、参数、输出 schema、baseline 数据或 compare gate，除非报告明确说明这是独立实验。
- Windows 不作为 Psi 性能权威；远端 Linux 才是性能判断来源。
- 每个候选必须走完整链路：patch -> remote build -> parquet compare -> timing -> 记录 -> verdict
- compare 不过，不允许看 timing 结论。
- 不允许 detached 黑跑；长时间运行必须有可见状态文件和日志。

必须实现或补齐的能力：
1. profile_to_candidates：从 profile、hotspots、perf/log、历史 attempts、cooldown 中生成候选队列。
2. patch_queue：每个候选 patch 必须可保存、可回放、可撤销。
3. neutral_pool.tsv：compare 过、未明显变慢、但单点没有赢过噪声带的候选必须保留。
4. NOISY_PENDING：单个候选遇到 noisy，不 accepted、不 rejected，保留 patch 和 retry 条件。
5. neutral stack：自动把兼容的 neutral 小改组合起来，跑 stack build/compare/timing。
6. auto_loop：自动循环执行候选，直到 accepted、收敛、预算结束、no targets、远端失败或用户停止。
7. status surface：每个 run_dir 顶层必须写 run_state.json、heartbeat.json、profile.tsv、hotspots.tsv、attempts.tsv、cooldown.tsv、neutral_pool.tsv、timing_history.tsv、logs/、patches/、reports/、charts/

候选来源必须分三条 lane：
1. evidence lane：来自 profile / hotspots 的高证据候选。
2. insight lane：模型提出的小而窄的 Class A / cache-locality 候选，即使它不在最大热点。
3. combination lane：correctness pass 的低风险 neutral 候选组合验证。

执行流程：
1. 检查当前任务、当前 worktree、远端连接、已有 run artifacts。
2. 建立或刷新当前 control 分布，记录样本、median、MAD、IQR、stdev、range。
3. 读取 profile.tsv、hotspots.tsv、attempts.tsv、cooldown.tsv、neutral_pool.tsv、timing_history.tsv。
4. 生成候选队列，覆盖 evidence / insight / combination 三条 lane。
5. 对每个候选写清楚 hypothesis、target、expected effect、semantic risk、touched files。
6. 在 worktree 中生成或应用 patch。
7. 跑远端 build。
8. build 通过后跑 parquet compare。
9. compare 通过后跑 timing。
10. 记录 raw samples，不只记录一个秒数。
11. 计算 median、MAD、IQR、stdev、range、sample_count、delta。
12. 写 attempts.tsv 和 timing_history.tsv。
13. 判定 accepted / neutral / rejected / NOISY_PENDING。
14. 更新 run_state.json、heartbeat.json、cooldown.tsv、neutral_pool.tsv。
15. 如果有足够 neutral 候选，尝试 neutral stack。
16. 每轮 batch 后更新报告或报告草稿。
17. 检查停止条件；不满足就继续下一轮。

停止规则：
- 出现真正 accepted
- 达到预算上限
- 连续无有效候选且收敛
- 所有候选进入 cooldown / no targets
- 远端不可用或 compare/build 失败
- 我手动要求停止

输出要求：
- 每轮给出简短状态
- 最终给出结构化报告
- 报告必须说明：做了什么、跑了什么实验、结果如何、为什么没赢、下一步怎么改
- 不要只给结论，要给可复盘的证据路径
```
