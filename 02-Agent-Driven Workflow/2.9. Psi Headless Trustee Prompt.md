# Psi Headless Trustee Prompt

This file is only a launch prompt.
Do not add long-term policy here.

## Long-Run Prompt

```text
你现在进入 Psi 长期 headless 自动优化托管模式。

Active task:
.trellis/tasks/05-10-psi-headless-optimization-v2

当前目标：
把 Psi headless 自动优化循环跑成完整态：先修证据链，再持续找候选、验证候选、保存 neutral、组合 stack，直到 accepted / 预算结束 / 收敛 / no targets / 远端失败 / 用户停止。

先决条件：
1. 先检查本地 HFT-wf 是否已经包含证据链修复：
   - comparison_summary.json 区分 old_control / candidate / updated_baseline
   - run_state.json 中 timing_status 与 accepted verdict 一致
   - patch_manifest.json 记录 patch snapshot 状态、changed files、dirty risk
   - 报告能展示 comparison_summary，而不是把更新后 baseline 和旧 control 混在一起
2. 如果这些能力还没有生效，先同步或修复 harness，再跑实验。
3. 不要直接进入新优化搜索来掩盖证据链问题。

硬约束：
- 远端 Linux 是唯一性能权威。
- Windows timing 不能作为 accepted 证据。
- 不允许修改 factor set、参数、输出 schema、baseline 数据或 compare gate 来让结果更好看。
- 不允许推送远端业务仓库。
- 不允许 detached 黑跑；长任务必须有 run_state.json、heartbeat.json、logs/ 和报告。
- 每个候选必须走：patch -> remote build -> parquet compare -> timing -> record -> verdict。
- compare 不过，不能看 timing 结论。
- 没有 patch snapshot 或 patch_manifest 的优化不能进入 accepted，只能进入 evidence_risk / blocked。
- 远端 worktree dirty 时，必须记录 dirty risk、changed files、candidate touched files，并说明 accepted 是否可复盘。

必须维护的 artifacts：
- run_state.json
- heartbeat.json
- comparison_summary.json
- profile.tsv
- hotspots.tsv
- attempts.tsv
- patch_queue.tsv
- neutral_pool.tsv
- retry_conditions.tsv
- cooldown.tsv
- timing_history.tsv
- patches/current_worktree.patch
- patches/patch_manifest.json
- logs/
- reports/
- charts/ 如果有图

候选来源分三条 lane：
1. evidence lane：来自 profile / hotspots / perf / logs 的高证据候选。
2. insight lane：模型提出的小而窄的 Class A / cache-locality 候选，即使不在最大热点。
3. combination lane：compare pass 且低风险的 neutral 候选组合验证。

执行循环：
1. 读取当前 run artifacts 和历史 timing_history。
2. 建立或刷新当前 control 分布，至少记录 sample_count、samples_ms、median、MAD、IQR、stdev、range。
3. 生成候选队列，覆盖 evidence / insight / combination。
4. 每个候选写清 hypothesis、target、expected effect、semantic risk、touched files、patch path。
5. 应用 patch，并保存可复盘 patch snapshot；如果 snapshot 失败，记录 evidence_risk，不能 accepted。
6. 远端 build。
7. build 过后 parquet compare。
8. compare 过后 timing。
9. 写 attempts.tsv、timing_history.tsv、comparison_summary.json。
10. 判定 accepted / neutral / rejected / NOISY_PENDING / evidence_risk。
11. neutral 进入 neutral_pool.tsv，不要丢。
12. NOISY_PENDING 保留 retry condition，不要全局停止。
13. 有兼容 neutral 时尝试 stack。
14. 每轮更新报告。
15. 检查停止条件；不满足就继续下一轮。

accepted 规则：
- build pass
- compare pass
- 同一 harness / 同一 host / 同一 gate 下 timing 明确赢过当前 control 分布
- comparison_summary.json 明确记录 old_control、candidate、updated_baseline
- patch snapshot 或 patch_manifest 足够复盘
- 报告说明 baseline 是否更新

neutral 规则：
- compare pass
- 没有明显变慢
- 单点收益不足以越过噪声带
- 必须进入 neutral_pool.tsv
- 之后优先尝试 compatible stack

NOISY 规则：
- NOISY 是候选级暂停，不是全局罢工。
- 单个候选 noisy 时标记 NOISY_PENDING，记录 retry condition。
- loop 必须继续尝试其他候选、其他 lane、profile refinement 或 neutral stack。
- 只有 control 分布连续多轮不可用，才允许全局 noise stop。

停止规则：
只有以下情况能停止整个长期 run：
- 达到 max-hours
- 达到 max-iterations
- 达到 max-candidates
- no_targets：所有 lane 都为空、blocked 或 cooldown
- convergence_proven：样本足够且数学模型显示剩余期望收益低于阈值
- remote_failed：远端无法 build/compare/timing，重试后仍不可恢复
- repeated_infra_failure：基础设施问题导致证据不可用
- 用户明确要求停止

不要因为以下情况停止：
- 一个候选 failed
- 一个候选 neutral
- 一个候选 NOISY
- 一个 patch 想法失败
- 一轮没有 accepted
- profile 不够细

如果出现 accepted：
- 如果启动参数是 first-accepted stop，可以停并报告。
- 如果目标是“长期榨干”，不要默认停；先完成 accepted artifact 审计，再继续下一轮，优先看剩余 hotspot 和 neutral stack。

每轮输出：
- 简短状态
- run_dir
- 当前 candidate
- build / compare / timing verdict
- accepted / neutral / rejected / NOISY_PENDING / evidence_risk counts
- latest report path
- next action

最终输出：
- final report path
- run_dir
- accepted count
- neutral count
- rejected count
- NOISY_PENDING count
- evidence_risk count
- final control median
- baseline 是否更新
- stop reason
- changed files
- artifact paths
- 是否还能继续榨，以及下一条最有希望的 lane

如果没榨出新 accepted：
不要只说“没有优化”。必须说明：
- 是候选质量问题、profile 粒度问题、噪声问题、gate 覆盖问题、patch 复盘问题，还是 harness 问题
- 已经如何改进流程
- 下一轮最有希望的方向是什么

现在开始：
先做证据链 audit；audit 通过后继续长期 headless 自动优化循环。
```

## Use

Copy the prompt above into the trustee agent launch path.

For an overnight run, prefer setting explicit budget values in the user message:

- `max-hours`
- `max-iterations`
- `max-candidates`
- whether first accepted should stop or continue
