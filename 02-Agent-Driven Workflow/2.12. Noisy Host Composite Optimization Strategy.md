# Noisy Host Composite Optimization Strategy

## 0. Why This Document Exists

The Psi headless optimization loop has reached a practical measurement limit:
many `readParquet` candidates pass build and compare, show positive median
speedup, and still remain `NOISY_PENDING` because the devbox has persistent
runtime jitter.

Continuing to run single-field micro-optimizations under the same gate is now
low leverage. It creates more positive-but-noisy records without producing a
clean baseline candidate.

This document changes the strategy:

```text
single micro candidate -> composite stack / stage-level candidate
hard accepted/noisy split -> promotion candidate under noisy host
per-candidate retry loop -> quiet-window retry or dedicated validation host
```

The goal is not to weaken correctness. Build and compare remain mandatory. The
goal is to stop treating a permanently noisy machine as if it were a reliable
microbenchmark host.

---

## 1. Observed Psi Pattern

The current Psi ledger contains several candidates with this shape:

```text
build_status       pass
compare_status     pass
median_delta_ms    positive
noise_flag         NOISY
timing_verdict     NOISY_PENDING
accepted           no
```

Examples from the current run family:

- `skip_unused_market_strings`
- `skip_unused_book_volume_assignment`
- `skip_unused_amount_assignment`
- `skip_unused_preclose_assignment`
- `int_code_set_first_pass`
- `skip_sort_pass_when_ordered`
- `reuse_bar_index_template`

This is not evidence that every patch is good enough to promote. It is evidence
that the current host can detect direction more often than it can prove
acceptance for small changes.

The loop should therefore stop spending most of its budget on immediate
single-candidate retries.

---

## 2. Research Signals

Recent work points in the same direction:

- Code efficiency benchmarks such as Mercury and ECCO emphasize that functional
  correctness and efficiency must be evaluated together; correctness alone is
  not enough, but efficiency measurements are also fragile across settings.
  Sources:
  - https://arxiv.org/abs/2402.07844
  - https://arxiv.org/abs/2407.14044
- Comparative work on LLM code optimization shows that LLMs can produce useful
  speedups but also incorrect code, so automated verification remains central.
  Source: https://arxiv.org/abs/2406.12146
- Agentic optimization systems such as Autocomp and CompilerGPT use structured
  iteration with correctness/performance feedback rather than one-shot advice.
  Sources:
  - https://arxiv.org/abs/2505.18574
  - https://arxiv.org/abs/2506.06227
- Superoptimization work shows that correctness-preserving optimization can
  benefit from iterative refinement and best-of-N style search, but the reward
  must combine correctness and speed, not speed alone.
  Source: https://arxiv.org/abs/2505.11480
- Cloud and shared-environment benchmarking studies confirm that variability can
  slow convergence and can make tuning results less transferable when the
  measurement environment is not representative.
  Sources:
  - https://www.microsoft.com/en-us/research/publication/performance-roulette-how-cloud-weather-affects-ml-based-system-optimization/
  - https://arxiv.org/abs/2504.11826

The practical translation for Psi is:

```text
keep automated correctness gates strict;
make performance decisions at the right granularity;
avoid pretending that a noisy shared devbox can cleanly accept tiny wins.
```

## 3. Strategy Shift

### 3.1 Composite Patch To Increase Signal

Single changes such as removing one assignment can be smaller than host noise.
When multiple independent, compare-passing, same-direction changes exist, the
next experiment should combine them into one explicit stack.

Example stack candidate:

```text
row_loop_unused_materialization_stack

members:
  skip_unused_market_strings
  skip_unused_book_volume_assignment
  skip_unused_amount_assignment
  skip_unused_preclose_assignment
```

Rules:

- The stack is an explicit composite experiment, not a way to bypass ledger
  gates.
- The agent must produce one coherent final C++ patch.
- The patch must not apply old patches blindly if their hunks conflict.
- The patch must not change `_columns`, output schema, compare config, or factor
  set.
- The patch must not use projection-prune/manual-remap.
- Build and compare must pass before timing is interpreted.

Why this is better:

```text
micro candidate: 300-800 ms signal, easily buried by devbox jitter
stack candidate: multiple seconds of signal, more likely to beat noise
```

### 3.2 Promotion Candidate Under Noisy Host

`NOISY_PENDING` should continue to block automatic acceptance. However, repeated
positive results under noise should not be treated the same as an unproven idea.

Introduce a non-promoting review state:

```text
PROMOTION_CANDIDATE_UNDER_NOISY_HOST
```

Use it only when all of the following are true:

- build passes
- compare passes
- at least two independent runs are positive, or one stack run has a large
  positive effect
- confidence interval is fully positive or materially away from zero
- `noise_flag=NOISY` is the only reason the harness refuses acceptance

This state means:

```text
do not auto-merge;
do not update baseline;
prepare human review or dedicated-host validation.
```

It is a parking state for strong evidence under a bad measurement environment.

### 3.3 Dedicated Validation Host Or Quiet Window

If the devbox has persistent load, it should not be the final authority for
small or medium performance wins.

Preferred validation environment:

- no other `PsiTraderRunner`
- stable CPU frequency governor
- isolated CPU set if possible
- fixed dataset and config
- repeated control samples before candidate runs
- same binary build mode for baseline and candidate

If this environment is unavailable, the loop should:

- keep single candidates in quiet-window retry only
- run explicit composite stacks when enough compatible positive candidates exist
- emit `PROMOTION_CANDIDATE_UNDER_NOISY_HOST` instead of pretending acceptance
  is possible

### 3.4 Change The Experiment Unit

The loop should stop spending its main budget on tiny single-line candidates
once a scope has produced several positive-noisy records.

Preferred units after that point:

1. Composite stacks of compatible compare-passing work reductions.
2. Stage-level optimizations, such as `handlerData.timestamp` or
   `write.generate_table`.
3. Dedicated-host validation of already discovered positive candidates.

Micro-candidates are still useful as discovery probes. They should not remain
the primary acceptance unit under a persistently noisy host.

## 4. Harness Policy

### 4.1 Ledger Classification

Keep the current ledger classes:

```text
positive_noisy       -> quiet-window retry only
negative_noisy       -> no positive retry
compare_fail         -> no timing retry
blocked/risky        -> do not generate
```

Add these stack-oriented classes:

```text
stack_prepared
stack_compare_fail
stack_positive_noisy
promotion_candidate_under_noisy_host
```

### 4.2 Stack Candidate Contract

A stack candidate record must include:

```text
candidate_id
stack_members
stack_reason
conflict_resolution_summary
touched_files
forbidden_surface_check
build_status
compare_status
timing_verdict
promotion_state
```

The stack is valid only if:

- each member has prior compare-pass evidence, or the combined patch is small
  enough to reason about directly
- the final patch passes boundary checks
- compare passes on the final stack

### 4.3 Acceptance Matrix

| Result | Action |
|---|---|
| build fail | `stack_build_fail`; stop stack |
| compare fail | `stack_compare_fail`; do not time |
| positive and clean noise gate | stop for human confirmation before promote |
| positive but noisy with large effect | `PROMOTION_CANDIDATE_UNDER_NOISY_HOST` |
| negative | `stack_negative`; do not retry |
| infra failure | retry only according to infra policy |

Automatic baseline update remains forbidden unless explicitly approved after a
clean accepted result.

## 5. Next Psi Move

The next high-value experiment is not another row-loop single-field deletion.
It is:

```text
explicit composite stack:
  skip_unused_market_strings
  skip_unused_book_volume_assignment
  skip_unused_amount_assignment
  skip_unused_preclose_assignment
```

Run shape:

```text
prepare one coherent patch
local materialization smoke
remote build
parquet compare
m12 paired timing
ledger classification
```

If it passes compare and remains noisy but strongly positive, classify it as
`PROMOTION_CANDIDATE_UNDER_NOISY_HOST`, not `accepted`.

If even the composite stack cannot rise above the noisy gate, the project should
stop trying to auto-accept on this devbox and move to dedicated-host validation
for any promotion decision.

## 6. Prompt Boundary For Future Agents

Use this when the loop starts wasting time on more micro-candidates:

```text
The current host is persistently noisy. Stop immediate single-candidate retries.
Use micro-candidates only as discovery probes. For acceptance evidence, prefer
explicit composite stacks or stage-level candidates. Keep build and compare
strict. Do not accepted noisy. Use PROMOTION_CANDIDATE_UNDER_NOISY_HOST for
strong positive evidence that cannot pass the host noise gate.
```
