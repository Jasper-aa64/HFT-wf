# Psi Automatic Optimization Loop Design

## 0. Document Boundary

This document defines the system design for the Psi automatic optimization loop.
It is the architecture contract for the harness.

It deliberately does not contain:

- current baseline values
- per-run experiment results
- the copy-paste runtime prompt
- an open-ended roadmap of unfinished ideas

Related documents:

- `2.5. Psi Optimization Evidence Boundary.md`: what counts as valid evidence.
- `2.8. Psi Optimization Inventory and Roadmap.md`: what is done, under-modeled, and next.
- `2.9. Psi Headless Trustee Prompt.md`: the prompt used to launch a headless trustee agent.

## 1. Goal

The loop turns Psi optimization from ad hoc candidate testing into a bounded,
auditable control system:

```text
profile -> candidate queue -> patch worker -> remote gate -> attempts/cooldown/run_state -> convergence decision -> report
```

The loop must solve four problems:

- find and verify the next optimization candidates automatically
- avoid greedy-only optimization of the largest hotspot
- make noise, convergence, and stop decisions explicit
- leave enough artifacts that the run can be reviewed later without chat context

The normal long-running scenario is:

```text
start headless run -> bounded overnight loop -> review report, artifacts, and verdicts the next day
```

## 2. Contract-Complete v1

The first usable version should be contract-complete, not toy-complete.

That means the first version may keep candidate generation simple, but the long-term
contract must already be present:

```text
run_state.json
heartbeat.json
profile.tsv
hotspots.tsv
attempts.tsv
cooldown.tsv
neutral_pool.tsv
timing_history.tsv
patches/
logs/
reports/
charts/
remote build / compare / timing gate
explicit stop reason
resume-friendly state
```

The v1 strategy may stay narrow:

- use three candidate lanes instead of a full bandit optimizer
- let Codex workers generate patches instead of making a script write code alone
- support business-stage profile plus optional `perf` evidence before full code-point attribution
- run bounded serial batches before large parallel search

The point of v1 is to make later strategy upgrades possible without rewriting the evidence contract.

## 3. Authority Model

The evaluator is more important than the agent.

An agent may propose candidates, but only the gate may declare success:

```text
remote build -> parquet compare -> no_compare timing -> attempt verdict -> optional bundle audit
```

Accepted, neutral, rejected, noisy, budget stop, and convergence all must be
derived from recorded artifacts, not from conversation memory.

Remote Linux is the performance authority. Windows is only a control/reporting
surface for this workflow.

## 4. Candidate Lanes

The loop uses three lanes so profile guidance does not become profile greed.

### 4.1 Evidence Lane

Evidence lane candidates come from `profile.tsv`, `hotspots.tsv`, `perf`, logs,
or repeated historical costs.

The target score may be written as:

```math
s = c \times o \times r \times l
```

where:

- `c`: observed cost
- `o`: ownership confidence
- `r`: correctness safety
- `l`: locality

This score is only for ordering evidence-lane targets. It must not suppress
insight-lane or combination-lane candidates.

### 4.2 Insight Lane

Insight lane candidates are small, local, low-risk ideas proposed from code
understanding:

- redundant work removal
- cache/locality improvements
- avoiding repeated conversions
- eliminating repeated scans

They must still pass the same build, compare, and timing gates.

### 4.3 Combination Lane

Combination lane validates safe neutral candidates as a stack.

A single neutral candidate does not update baseline. A compatible neutral stack
may be accepted only when the whole stack passes compare and wins over the
control distribution.

## 5. Candidate Lifecycle

Each candidate must have a durable record:

```text
candidate_id
lane
hypothesis
target
expected_effect
semantic_risk
touched_files
patch_path
build_result
compare_result
timing_summary
verdict
retry_condition
```

The required execution order is:

```text
write/apply patch
remote build
parquet compare
timing
record raw samples and statistics
decide verdict
update state
keep/revert/stack
```

Timing must not be used as acceptance evidence if compare fails.

## 6. Verdicts

### accepted

The candidate or stack passed compare and clearly beat the current control
distribution under the same harness, host, gate, and compatibility group.

### neutral

The candidate passed compare and did not clearly regress, but its gain was not
strong enough to beat the noise model. It goes to `neutral_pool.tsv`.

### rejected

The candidate failed build, failed compare, clearly regressed, had unacceptable
semantic risk, or was not covered by the active gate.

### NOISY_PENDING

The candidate may be valuable, but the sample noise is too high for a reliable
accepted/rejected decision. This is a candidate-level state, not a global stop.

It must keep a retry condition.

## 7. Stop Rules

The loop may stop only for explicit run-level reasons:

- `accepted` with first-accepted mode enabled
- `budget_stop` for max-hours / max-iterations / max-candidates exhaustion
- `no_targets`
- `convergence_proven`
- `remote_failed`
- user stop

The following are not global stop reasons:

- one candidate failed
- one candidate was neutral
- one candidate was noisy
- one profile was too coarse
- one iteration had no accepted candidate

## 8. Observable Headless Execution

Headless does not mean invisible.

Every long run must expose:

```text
heartbeat.json
run_state.json
current log
attempts.tsv append stream
latest report path
explicit stop reason
```

Detached remote execution is not allowed unless the run still has a visible
status surface and the user explicitly approved that mode.

### 8.1 Retention and Cleanup

The loop must not accumulate duplicate long-lived notes.

Keep the following at the run-root or task level:

- machine-readable state for the current run
- the latest report and links to older reports
- the current neutral pool, cooldown set, and attempts history
- a short summary of what changed this batch

Do not copy raw logs, charts, or PDFs into long-lived policy docs.
Do not move per-run medians or temporary baselines into `2.5`, `2.7`, or `2.8`.
When a task closes, archive the task directory and keep only the summary in the
workspace journal.

## 9. User-Facing Modes

### Headless Mode

Used for overnight or long-running experiments.

One batch should normally contain:

```text
profile refresh
up to 3 candidates
optional neutral stack or bundle audit
state/report update
continue/stop decision
```

### Interactive Mode

Used when the main thread is discussing strategy while a visible worker runs a
bounded batch.

Interactive mode may use a worker, but the worker must produce visible artifacts
and a terminal status.

## 10. What Belongs Elsewhere

- Experiment results belong in run artifacts and task execution notes.
- Measurement policy refinements and remaining gaps belong in `2.8`.
- Launch instructions and the trustee prompt belong in `2.9`.
- Evidence authority rules belong in `2.5`.
