# Psi Automatic Optimization Loop Design

## 0. 设计目标

这份文档定义 Psi 性能优化的自动 harness：

```text
profile -> candidate queue -> patch worker -> remote gate -> attempts/cooldown/run_state -> convergence decision -> report
```

它不是“agent 随便跑一夜”。它要解决四个问题：

- 自动找和验证下一批优化候选。
- 避免只盯最大热点的贪心搜索。
- 用数学化指标判断收敛、噪声和预算停止。
- 让长时间 headless 运行第二天可复盘，而不是只剩聊天记录。

最终主场景是：

```text
下班前启动 headless run -> 夜间自动执行有界循环 -> 第二天验收 md + pdf + artifacts
```

白天 interactive 模式仍然存在，但它是观察、调参和讨论模式，不是这套 harness 的唯一使用方式。

## 1. 第一版定位：contract-complete v1

这里不建议做一个“只要能跑就行”的低配版本。性能优化 harness 一旦缺少状态、退出规则、报告和恢复能力，很快会退化成黑箱实验。

更合适的第一版是：

```text
contract-complete v1
```

意思是：第一版就把长期契约做好，但把策略复杂度控制住。

v1 必须一次做好的部分：

```text
run_state.json
profile.tsv / hotspots.tsv / attempts.tsv / cooldown.tsv
patches/ logs/ reports/
remote build/compare/timing gate
mathematical stop classifier
headless heartbeat/status
performance report md + pdf
convergence charts
millisecond timing artifacts with seconds compatibility summaries
resume / stop reason
```

v1 可以先做窄的部分：

```text
候选生成先用三 lane 规则，不上复杂 bandit
worker patch 先由 Codex worker 完成，不把 patch 生成完全塞进脚本
profile 先支持业务阶段 + perf fallback，不追求全函数级自动归因
headless 先串行执行 batch，不做大规模并行搜索
```

这样第一版就是可复用骨架。后续增强的是策略，不是补地基。

## 2. 调研依据

外部资料只用于抽象自动优化系统的形状，不证明任何 Psi 候选安全或变快。Psi 的证据仍只来自远端 Linux build、parquet compare、timing、bundle audit。

| 来源 | 可借鉴点 | 对 Psi 的限制 |
|---|---|---|
| [Karpathy autoresearch](https://github.com/karpathy/autoresearch) | 用 `program.md` 和固定 evaluator 约束 agent；agent 编辑有边界文件，评价器决定 keep/revert。 | 它是 ML 研究循环，不是 C++ 低延时管线。 |
| [AutoKernel](https://github.com/RightNow-AI/autokernel) / [arXiv](https://arxiv.org/abs/2603.21331) | profile bottleneck、bounded kernel、固定 benchmark、correctness first、keep/revert、按 impact 换目标。 | 它优化 GPU kernel，Psi 有 CPU、Arrow/Parquet、I/O 和业务语义。 |
| [PyTorch KernelAgent](https://pytorch.org/blog/kernelagent-hardware-guided-gpu-kernel-optimization-via-multi-agent-orchestration/) | `Profile -> Diagnose -> Prescribe -> Orchestrate -> Explore -> Measure`，阶段间用结构化输出。 | GPU profiler 和 kernel benchmark 不能直接替代 Psi 业务 gate。 |
| [Meta KernelAgent](https://github.com/meta-pytorch/KernelAgent) | 每轮有 run artifacts、worker session、benchmark result，便于复验。 | 只借鉴 artifact 纪律，不照搬 GPU 结构。 |
| [OpenTuner](https://github.com/jansel/opentuner) / [paper](https://classes.cs.uchicago.edu/archive/2017/fall/32001-1/papers/opentuner.pdf) | 不押注单一搜索策略；多策略共享结果数据库，按表现分配预算。 | Psi 候选需要改代码，不是纯参数空间。 |
| [Google Benchmark user guide](https://github.com/google/benchmark/blob/main/docs/user_guide.md) | warmup、repetition、aggregate statistics 是性能结论的基础。 | Psi 当前是全链路 runner，不是微基准。 |
| [Georges, Buytaert, Eeckhout, OOPSLA 2007](https://biblio.ugent.be/publication/417084) | 性能差异需要统计严谨性；没有置信区间，观测差异可能只是噪声。 | Psi wall-clock 粗粒度更需要噪声地板。 |
| [Group sequential design](https://www.stata.com/features/group-sequential-designs/) | 多次中途观察要预设 stopping boundary，避免临时 peeking。 | Psi 借鉴纪律，不做临床试验级复杂校正。 |
| [Linux perf manual](https://man7.org/linux/man-pages/man1/perf-report.1.html) | `perf record/report` 可以补函数级 CPU profile。 | perf 是补充证据，不能替代 parquet compare。 |

## 3. 核心原则

### 3.1 Evaluator first

agent 可以提出候选，但不能宣布成功。

```text
remote build -> parquet compare -> no_compare timing -> bundle audit -> attempts.tsv
```

成功、失败、中性、噪声、收敛都必须由 gate 和 artifact 记录。

### 3.2 Stateful, not chat-memory

自动循环必须把状态写到文件：

```text
run_state.json    当前状态、预算、退出原因、统计指标
profile.tsv       profile 原始或归一化阶段耗时
hotspots.tsv      排名后的热点证据
attempts.tsv      候选、patch、gate、样本、verdict
cooldown.tsv      blocked/cooldown/revisit 条件
neutral_pool.tsv  可组合的低风险 neutral 候选
```

没有这些文件，就不是自动 harness，只是长聊天。

### 3.3 Profile-guided, not profile-greedy

profile 排名只服务 evidence lane，不垄断搜索。自动候选队列必须同时包含：

```text
evidence lane:
  profile/hotspots 指向的高证据目标。

insight lane:
  模型提出的小而窄的 Class A / cache-locality 候选，即使不在最大热点。

combination lane:
  correctness pass 的低风险 neutral 候选组合验证。
```

### 3.4 Observable headless

headless 不是 invisible。允许跑很久，但必须可观察、可中断、可复盘：

```text
heartbeat.json
run_state.json
current.log
attempts.tsv 持续追加
明确 stop reason
最终报告 md + pdf
```

禁止没有状态面的 `nohup`、`tmux`、`screen` 式黑箱远端任务，除非用户显式批准并指定 status surface。

## 6. Candidate queue

### 6.1 Evidence lane

Evidence lane 使用期望收益 / ROI 目标分：

```math
\text{score_evidence} =
\frac{E[\Delta T] \times p_{owned} \times p_{safe} \times p_{gate} \times p_{local}}
{\text{cost_attempt}}
+ \lambda \times \text{uncertainty}
```

其中：

```text
E[Delta T]: 基于 profile 和历史 attempts 估计的期望运行时间下降量
p_owned: 该成本是否落在我们可修改代码中的概率 / 置信度
p_safe: 语义安全概率
p_gate: 当前 compare/gate 是否覆盖候选语义的概率 / 置信度
p_local: locality、rollback、attribution 是否清晰可靠的概率 / 置信度
cost_attempt: implementation + build + compare + timing 的尝试成本
uncertainty: 探索奖励，避免 evidence lane 退化成纯贪心 exploitation
lambda: 探索权重
```

旧的乘积式 `c × o × r × l` 只适合作为早期粗略启发式，不应该成为长期 driver policy。

`score_evidence` 只用于排序 evidence lane。Insight lane 和 combination lane 继续保留独立预算，不能被 evidence lane 分数压制。

### 6.2 Insight lane

Insight lane 允许模型提出 profile 最大项之外的小候选，但每个候选必须声明：

```text
hypothesis
touched files
expected affected stage
cache/locality mechanism
semantic risk
candidate class
rollback plan
```

典型候选：

```text
复用已计算值
减少重复字符串构造
减少重复 map lookup
减少热路径分配
改善连续访问
减少对象热字段跨度
```

### 6.3 Combination lane

Combination lane 解决 neutral 候选“积少成多”的问题。

规则：

```text
只纳入 compare pass 的低风险 neutral 候选。
不纳入有语义 caveat 的 neutral 候选。
每叠一个候选先 compare。
最终组合跑 bundle audit。
组合不赢整体回滚。
```

## 7. Gate policy

候选必须按顺序过 gate：

```text
1. patch applies
2. remote build pass
3. parquet compare pass
4. timing samples collected
5. attempt verdict written
6. bundle audit when required
```

Class A：

```text
correctness 必须过
性能记录但不被单个 noisy 样本一票否决
promotion 前需要 bundle drift audit
```

Class B：

```text
correctness 必须过
需要统计证据或 bundle evidence
无收益默认不进主线
```

Neutral stack：

```text
每步 correctness pass
组合后 bundle audit
无 aggregate gain 则整体回滚
```

## 8. 收敛与停止模型

### 8.1 收益定义

runtime 越低越好。第 `i` 个候选的收益：

```math
\delta_i = T_{control} - T_{candidate}
```

`\delta_i > 0` 表示变快。`T_control` 和 `T_candidate` 必须来自同一 harness、同一 active gate、同一 warm/cold 条件。

### 8.2 噪声地板

每个 harness 配置 material-improvement threshold：

```math
\epsilon = \max(\epsilon_{abs}, k \times \sigma_{control})
```

coarse wall-clock 秒级计时默认：

```text
epsilon_abs = 1s
k = 2
```

如果换成更稳定的阶段计时或微基准，必须重新标定 `epsilon_abs` 和 `k`。

### 8.3 置信区间停止

对 target 或 lane 最近 `n` 个候选的 `delta`，计算期望收益的 95% 上置信界：

```math
UCB_{95}(E[\delta]) \le \epsilon
```

只有满足最小样本数并且满足上式，才能写：

```text
convergence_proven
```

候选和 control 都有重复样本时，记录 Welch CI / Welch t-test，而不是只比较均值。

### 8.4 边际收益斜率

对 accepted 累计收益曲线估计斜率：

```math
slope = \frac{\Delta cumulative\_improvement}{\Delta accepted\_index}
```

若最近 `m` 个 accepted 点满足：

```math
|slope| \le \epsilon_{slope}
```

或最近 accepted deltas 全部低于 `epsilon`，可标记：

```text
plateau
```

这是辅助信号，因为 accepted 点可能很稀疏。

### 8.5 Lane 预算模型

对每个 lane 使用简单 Beta 先验：

```math
p_{accept} \sim Beta(1,1)
```

若该 lane 已有 `a` 个 accepted，`r` 个 rejected/neutral：

```math
E[p_{accept} | a,r] = \frac{a+1}{a+r+2}
```

当：

```math
E[p_{accept}] \times estimated\_expected\_win \le cost\_threshold
```

该 lane 进入 cooldown，退出原因写：

```text
lane_cooldown_budget
```

这只说明继续投预算不划算，不证明不存在优化机会。

### 8.6 噪声停止

如果 `range`、`stdev` 或 coefficient of variation 超过 harness 阈值：

```text
NOISY
```

`NOISY` 不允许接受、拒绝或宣布收敛。应换时段、增加样本、提高计时精度，或先降低环境噪声。

### 8.7 退出状态

```text
convergence_proven       UCB_95(E[delta]) <= epsilon 且样本数达标
plateau                 accepted 累计收益斜率低于 epsilon_slope
lane_cooldown_budget     Beta(1,1) 模型显示继续投预算不划算
NOISY                   噪声超过阈值，暂停性能判定
bundle_regression        bundle audit regression
budget_stop              max-hours / max-iterations / stall-limit 触发
no_targets               all high-score targets are cooldown or blocked
correctness_failed       compare 或语义 gate 失败
remote_failed            远端不可用或 build 环境失败
```

`stall-limit` 只能触发：

```text
budget_stop
```

不能触发：

```text
convergence_proven
```

## 9. 报告与收敛图

每次 headless 或 interactive batch 结束都要生成性能优化报告：

```text
C:\Users\liangjunming\Desktop\work\日报（中文）\<date>\<date> 性能优化报告.md
C:\Users\liangjunming\Desktop\work\日报（中文）\<date>\<date> 性能优化报告.pdf
```

注意：父目录只是用户的日期工作区，报告标题和正文都叫“性能优化报告”，不是日报。

报告至少包含：

```text
简短总结
control/best/current 状态
accepted / neutral / rejected 表
profile top hotspots
stop reason
convergence metrics
runtime convergence chart
convergence decision chart
```

### 9.1 runtime convergence chart

展示性能曲线：

```text
x: accepted step 或 candidate step
y: runtime median
markers: baseline / accepted / neutral / rejected
annotation: each accepted delta
```

用途：说明 runtime 是否持续下降，以及收益是否变小。

### 9.2 convergence decision chart

展示停止依据：

```text
x: candidate index 或 lane iteration
y: delta = T_control - T_candidate
horizontal line: epsilon
error bar: 95% CI 或 UCB
marker color: accepted / neutral / rejected / NOISY
annotation: stop reason
```

用途：说明为什么是 `convergence_proven`、`budget_stop`、`NOISY` 或 `lane_cooldown_budget`。

没有第二张图时，报告只能说“趋势看起来收敛”，不能声称数学意义上的 `convergence_proven`。

## 10. 自动循环

```text
initialize run_state
verify remote control

while budget remains:
    write heartbeat
    profile = produce_or_load_profile()
    hotspots = rank_hotspots(profile)
    queue = build_candidate_queue(hotspots, attempts, cooldown, neutral_pool)

    batch = pick_contract_v1_batch(queue)

    for candidate in batch:
        write candidate hypothesis
        patch = run_patch_worker(candidate)
        result = remote_build_compare_timing(patch)
        append_attempt(result)
        update_neutral_pool(result)
        update_cooldown(result)

        if result.correctness_failed:
            revert_candidate()
            stop("correctness_failed")

    if bundle_required:
        audit = run_bundle_audit()
        if audit.regressed:
            stop("bundle_regression")

    metrics = compute_convergence_metrics(attempts, control_samples)
    write_charts(metrics)
    write_report()

    if metrics.noise_status == "NOISY":
        stop("NOISY")
    if metrics.min_samples_met and metrics.ucb95_expected_delta <= epsilon:
        stop("convergence_proven")
    if metrics.plateau:
        cooldown_lane("plateau")
    if metrics.lane_roi_below_threshold:
        cooldown_lane("lane_cooldown_budget")
    if no_targets:
        stop("no_targets")
    if budget_exhausted or stall_limit_reached:
        stop("budget_stop")

write_final_report()
```

## 11. contract-complete v1 实施边界

v1 要实现：

```text
scripts/psi_auto_optimize.py
scripts/psi_auto_status.py
run_state.json / heartbeat.json
profile.tsv / hotspots.tsv / attempts.tsv / cooldown.tsv / neutral_pool.tsv
headless mode
interactive mode
contract-v1 batch
remote build/compare/timing adapter
mathematical stop classifier
runtime_convergence.png
convergence_decision.png
性能优化报告 md/pdf
```

v1 暂不实现：

```text
复杂 bandit / Bayesian optimizer
多 worker 并行探索
完全脚本内自动生成 patch
自动修改未被 active gate 覆盖的因子
无状态 detached remote job
```

第一版的质量标准：

```text
即使没有找到新优化，也必须留下完整、可信、可复盘的自动优化证据。
```

## 12. 结论

Psi 自动优化的方向不是：

```text
最大热点 -> 改 -> 跑 -> 无限重复
```

而是：

```text
observable headless + fixed gate + three-lane candidate queue + mathematical stop model + convergence charts + dated performance report
```

第一版应该是 contract-complete v1：状态、gate、退出、图表、报告一次做好；候选生成策略先保守，后续再扩展。
